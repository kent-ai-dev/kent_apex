# RCE Storage and Scaling Architecture

## TL;DR

- **NOT a SQL database.** SQL is the wrong shape for nested combinator trees,
  top-k weight queries, and memoized output caches.
- The real store is a **hybrid of three things**: a graph DB for program
  lineage, a content-addressed blob store for program code, and a key-value
  cache for memoized outputs.
- **Inference latency stays roughly flat** as the library grows past a few
  thousand programs, because top-k pruning bounds the active working set.
- **Training latency grows linearly** in library size — that's what Modal
  sharding solves.
- **Memory grows logarithmically** in data trained on, because abstraction
  (PLAN.md V5) compresses recurring patterns into single primitives.

## Why not SQL

- Programs are **nested combinator trees**: `compose(branch(ngram-3, recur(ngram-2)), abstract(ngram-4, ngram-1))`. Decomposing this into relational tables is awkward and queries against the structure are slow.
- The hot query is **top-k by posterior weight**, not point lookup by ID. Heaps and sorted indices in KV stores beat B-trees for this.
- We need **content-addressing**: identical program structures should share storage. SQL doesn't natively dedupe this way; content-addressed object stores (git-style) do.
- Output caching needs **LRU eviction over millions of keys**. RocksDB / LMDB do this natively; SQL doesn't.

## The four stores

### 1. Program graph (LMDB or Neo4j)

Each program is a node:
```
program_id (uuid):
  name: str            # human-readable, e.g. "compose(ngram-3,ngram-2)"
  combinator: str      # primitive | compose | branch | abstract | recur
  length: float        # description length in bits (for Solomonoff prior)
  code_hash: str       # → code store
  parent_ids: [uuid]   # lineage
```

Edges are parent→child relationships. Adjacency lists in LMDB are fine; full graph DB only needed if you want to do graph queries like "shortest path between two programs."

**Hot queries:**
- get program by id (point lookup)
- get top-k by current weight (joined with weights store)
- get parents/children of program X (lineage)
- iterate all primitives (for fitting at training time)

### 2. Program code (content-addressed blob store)

A flat directory or LMDB keyed by SHA-256 of the program's structural representation. Two combinators that independently produce identical structures share one entry. This is git's object-store pattern, and it's perfect here.

```
code/
  ab/cd/abcdef0123...  # primitive: pickled UniformPrimitive
  12/34/123456789...   # combinator: pickled Composed(parent_a_id, parent_b_id, alpha)
```

For non-primitive programs we store *references* to parent IDs, not nested objects. This keeps each blob tiny (a few hundred bytes) and lets the graph store handle the lineage independently.

### 3. Weights (mutable LMDB)

Hot, frequently-updated, small per-entry:
```
program_id → log_posterior_weight (float64)
```

This is the only store written to during normal Bayesian updates. Atomic writes via LMDB transactions. Sharded by program_id hash for distributed training.

### 4. Output cache (LMDB with LRU)

The performance multiplier:
```
(program_id, context_hash) → predicted_distribution (Counter, gzipped)
```

Most programs are deterministic functions of recent context. If we cache their outputs by context hash, repeated contexts during generation get free hits. With an 8-byte context window, the cache space is bounded — most contexts in trained data are repeats of common patterns.

LRU eviction by access timestamp; cap total size at e.g. 4 GB.

## Inference latency math

Cost per generated byte ≈ `top_k * avg_program_eval_cost + cache_lookup_overhead`

With:
- top_k = 100 (programs that carry 99% of posterior mass)
- avg_program_eval_cost ≈ 10 µs (cached) or 1 ms (cold)
- cache hit rate ≈ 90% on common contexts

Total: ~10 ms per byte at million-program library scale, on a single CPU core. Roughly **constant in total library size** because top_k doesn't grow.

The latency floor is set by:
1. The number of programs we evaluate (top_k)
2. The cost of the slowest program in top_k (often a high-order n-gram lookup)
3. Cache miss rate

You can buy more speed by:
- Lowering top_k (at the cost of slightly worse predictions)
- Compiling primitives to C (n-grams via KenLM, repeat via memcmp)
- Parallelizing the top_k evaluations across cores (they're independent)

## Training latency math

Cost per training byte ≈ `library_size * avg_program_eval_cost`

Training updates ALL programs (not just top_k), because we need to discover when previously-low-weight programs start predicting well. This grows linearly in library size.

At a million programs and 100 µs per program eval: ~100 seconds per byte. That's catastrophic. Solutions:

1. **Subsample training updates.** Update on every Nth byte instead of every byte. PLAN.md `update_every` is already this.
2. **Cohort updates.** Sort programs into cohorts by recent weight; update high-weight cohorts every byte, low-weight cohorts every 10-100 bytes. Programs near the decision boundary still get prompt updates.
3. **Distributed across workers (Modal V8).** Shard the corpus, each worker holds its own library copy, periodic merge. Linear scaling within reason (merge costs eat into it).
4. **Prune aggressively during training.** Don't let library size grow past what you can afford. The Bayesian posterior naturally selects winners — kill losers.

## Memory footprint at scale

For a library trained on 100 GB of text:
- **Graph store:** ~100 MB (a million nodes with metadata)
- **Code store:** ~200 MB (deduplicated structures)
- **Weights:** ~10 MB (just floats)
- **Output cache:** 1-4 GB (capped, LRU)
- **N-gram primitive indices (KenLM-style mmap):** 2-10 GB depending on order

Total disk: ~10-20 GB. Resident RAM: 1-4 GB if you mmap everything.

For comparison: a 7B-parameter LLM is ~14 GB on disk and 14 GB resident. We're an order of magnitude smaller.

## Sharding for Modal (PLAN.md V8)

Each Modal worker gets:
- Read-only mount of the graph + code stores (small, replicated)
- Read-write mount of its own weight shard (partitioned by `hash(program_id) % N`)
- Local output cache

Workers train independently. Periodically (every N bytes, or every K seconds), a master node:
1. Reads all worker weight shards
2. Merges by **summing log-weights** for each program (Bayesian aggregation)
3. Renormalizes
4. Pushes merged weights back to workers

Communication cost: ~10 MB per merge per worker. Computation cost on master: O(library_size). Cadence: every minute is fine; every second is overkill.

For library growth (combinator generation): each worker proposes children locally; master collects proposals, deduplicates by content hash, assigns initial weights, broadcasts new programs back. New programs are rare events at scale (a few per second), so this is cheap.

## What we have today vs. what we need

**Today (seed):** single pickle file at `~/.rce_library.pkl`. Holds everything in one Python object. Fine up to ~10K programs, breaks down somewhere around 100K.

**V6 (HuggingFace ingestion):** time to migrate. Switch to LMDB for weights and output cache; keep code as a directory of pickles for now.

**V8 (Modal sharding):** add the master/worker merge protocol. Add KenLM as primitive backend for n-grams.

**V13 (self-modifying interpreter):** the interpreter itself becomes a stored program. The store needs to handle the case where the *interpreter* is being rewritten while running — copy-on-write semantics, atomic swap on accepted rewrites.

**V16 (provenance):** dedicate a provenance store. Append-only JSONL per program is fine; never delete provenance entries even when programs are pruned (we want to know they existed).

## Scaling past 100M programs — the production roadmap

The four-store design above is good to ~100M programs on serious hardware. Beyond that, several specific things break, and each has a known production-grade solution. This section enumerates them.

### What actually breaks at 100M+

1. **Top-k by weight becomes intractable.** You can't iterate the whole library every prediction to find the top 100. Sort-on-read is O(n log n); at a billion programs that's seconds per query.

2. **The output cache thrashes.** Context space grows combinatorially. Naive `(program_id, context) → distribution` maps blow past terabyte scale and hit rate collapses.

3. **All-program Bayesian updates become unaffordable.** Even subsampled, touching a billion programs per training byte is hopeless.

4. **Single-machine RAM** can't hold the working set. The library must be served distributed.

5. **Merge operations** (master node aggregating worker weight shards) become the bottleneck. A billion-row reduction every minute is a non-trivial pipeline.

6. **Combinator generation duplicates work across workers.** Without coordination, many workers independently propose the same children, wasting compute.

7. **The lineage graph itself** — billions of nodes with adjacency — outgrows single-node LMDB. Need partitioning or a real distributed graph engine.

### Tier system and the technologies that solve each

The right way to think about this is in tiers. Each tier specifies the library size it targets, what changes from the previous tier, and the tech we reach for. We don't jump straight to the trillion-program design — we step through, because each tier teaches us about the next.

#### Tier 1: ≤100M programs (single beefy machine + Modal workers)

**Status:** the design above covers this. LMDB stores, KenLM for n-grams, Modal for distributed training. Single-node serving.

**Cost:** ~$200/mo for a c7i-4xlarge equivalent + Modal training spend.

#### Tier 2: 100M – 1B programs (sharded production v1)

**Storage:** Replace LMDB with **FoundationDB**. FDB is the right tool here for several specific reasons: it gives ACID transactions across shards (critical for atomic library updates during combinator generation), it scales linearly to ~10B keys per cluster, and its layered design lets us put a graph layer (lineage) and a KV layer (weights, outputs) in the same cluster. Apple, Snowflake, and Wavefront all run FDB at billions of keys in production.

Alternative: **TiKV** (the storage engine under TiDB). Simpler, also production-tested, slightly weaker on cross-shard transactions. Pick FDB if you can stomach the operational complexity, TiKV if you want easier ops.

**Top-k weight queries:** This is where the change is biggest. Stop computing top-k on read. Instead maintain a **streaming top-k structure**:
- A **Count-Min Sketch with heavy hitters** keeps approximate top-k continuously updated as weights change. Constant memory, sub-millisecond queries, well-understood error bounds (Cormode & Muthukrishnan have the foundational papers).
- For exact top-k: a **min-heap of size k** maintained alongside the weights store, updated on every weight change. The heap fits in RAM at k=10,000 even for a billion-program library.

**Output cache:** Three tiers.
- **Hot tier: Redis** in-memory, ~100GB, holds the most-requested ~10M (program_id, context) pairs. Microsecond access.
- **Warm tier: RocksDB** on local NVMe, ~10TB, holds another billion entries.
- **Cold tier: S3** with lazy loading for the long tail. Some cache misses are fine — they just trigger a recompute, which is cheap because the program is right there.
- **Bloom filter** in front of the warm tier so we skip definite misses without paying the lookup cost.

**Bayesian updates:** Switch from "update all programs every Nth byte" to **stratified updates**:
- Top 0.1% (by weight): update every byte
- Top 1%: update every 10 bytes
- Top 10%: update every 100 bytes
- Long tail: update every 10K bytes, or only during periodic "deep updates"

This is mathematically defensible — programs in the long tail have weights so low they'd need many consecutive correct predictions to climb into top-k, and missing some updates just delays that climb without preventing it.

**Distributed compute:** Move from Modal alone to **Modal + Ray**. Ray gives us stateful actors — each actor holds a shard of the library and serves predictions for that shard. Modal handles bursty training jobs. Ray handles always-on serving. The two compose well; Modal can spawn Ray clusters.

**N-gram backend:** Beyond KenLM. At billion-n-gram scale, switch to **succinct data structures**: a Marisa-trie or a Roaring-bitmap-backed n-gram index uses ~3 bits per n-gram (vs ~30 bits for KenLM's mmap format). Microsoft and Google have published the techniques. Practical implementation: the `marisa-trie` Python lib, or write a thin C wrapper around the `succinct` C++ library.

**Sharding strategy for the lineage graph:** Partition by **program ID hash, then locality-preserve by lineage**. Initial assignment is by hash (uniform load), but during periodic compactions, programs are migrated so that parents and children tend to land on the same shard (cuts cross-shard adjacency lookups). This is graph partitioning à la METIS, run as a background job.

**Cost:** ~$5–15K/mo for the cluster (FDB, Redis, Ray, NVMe storage). Worth it past ~500M programs because algorithm-only optimizations have hit their ceiling.

#### Tier 3: 1B – 100B programs (production v2)

This is where it stops being "scale up the obvious things" and starts requiring **architectural** changes specific to the symbolic-AI substrate. LLMs don't need these — their parameters are fixed. We do.

**Hierarchical libraries.** The library becomes a *tree*, not a flat set. The root holds general-purpose programs; branches specialize by domain (text, code, math, dialogue) and sub-domain (Python, English, formal logic). Predictions route through the tree: at each level, only evaluate programs in branches whose context-signature matches the current input. This is the same structural insight that makes mixture-of-experts (MoE) work for LLMs — but applied to discrete programs instead of dense subnetworks, and dispatched by explicit context features rather than learned routing.

The router itself is a small program in the library. (Yes — this means the router can be self-modified by the same combinator process. Yes, that's a fixed point that needs careful handling.)

**Probabilistic execution.** Instead of running top-k programs deterministically, **sample a subset of top-k weighted by their posterior** and treat the result as an unbiased estimator of the full predictive distribution. This is importance sampling. It gives you 10–100× speedup at the cost of slightly noisier predictions, and the noise is bounded and characterizable.

**Lazy program instantiation.** Most programs in a tier-3 library are never queried. Don't even materialize their `predict` methods until something asks. Store them as serialized combinator structures; instantiate on demand; cache instantiated objects with LRU eviction. Saves orders of magnitude of RAM.

**Memoization at the combinator level.** A program `compose(A, B)` is a pure function of A's and B's outputs. Cache at every level of the combinator tree, not just at the top. Library-wide, this turns into a DAG memoization problem — well-understood in functional programming, well-suited to content-addressed storage. Every cached result is keyed by `(combinator_type, parent_output_hash_a, parent_output_hash_b)`. Identical computations across different programs hit the same cache entry.

**Storage: shift to a tiered architecture.**
- **Hot tier: Aerospike** for sub-millisecond hot-path KV. Aerospike's hybrid memory/SSD architecture is genuinely the best in class for this access pattern; it's what AdRoll, Kayak, and PayPal use for similar workloads.
- **Warm tier: ScyllaDB** for the bulk of the library. Cassandra-compatible, written in C++ on Seastar, deterministically faster than Cassandra. Handles a billion rows per node.
- **Cold tier: S3 + Apache Arrow / Parquet** for archival and bulk training reads. Programs that haven't been touched in months live here; pulling them is rare and slow but supported.

**Combinator generation across the cluster.** Workers proposing children must coordinate to avoid duplication. Use a **distributed bloom filter** of "structures considered in the last N hours" — workers check before proposing, master verifies on receipt. Hash-based deduplication catches structurally identical proposals; semantic deduplication (programs that compute the same function despite different structure) is harder and usually not worth solving.

**Distributed compute: Ray cluster + Kubernetes.** Modal alone isn't enough for always-on serving at this scale. Ray on Kubernetes gives you stateful actors with autoscaling. Modal still wins for bursty offline training jobs.

**The merge operation** becomes a real distributed reduce. Use **gradient-style aggregation** patterns from federated learning: hierarchical reduce trees, partial merges at intermediate aggregators, only the final merged top-k flows back to all workers. Borrowed from Horovod / NCCL conceptually, implemented on plain TCP because we're not doing matmul.

**Cost:** ~$100K/mo for a serious deployment. At this scale you're competing with mid-sized LLM inference costs, but with structurally different properties (auditable, calibrated, no hallucination). Whether it's worth it depends entirely on the BPB-vs-cost numbers from PLAN.md V11–V12.

#### Tier 4: 100B – 1T+ programs (research frontier)

We don't actually know if this works. Beyond ~100B programs, there are open research questions that PLAN.md V20 is supposed to surface. But here's the direction:

**Sparse structures over program space.** Don't store every program; store a *manifold* in program space defined by combinator dynamics, and instantiate programs lazily by traversing the manifold. The library becomes a generative process, not a collection. This is closer to how **continuous attractors** work in computational neuroscience — the library "is" the attractor landscape, and queries pull out specific basins.

**Geometric program embeddings.** Hash each program to a vector (by its prediction signature on a canonical context set), store in **FAISS** or **ScaNN** or **HNSW**. At query time, find the k programs whose embeddings are closest to the "query embedding" (the prediction profile we're looking for). This is approximate nearest neighbor at billion-vector scale and it's a solved problem — Facebook, Spotify, and Google all run trillion-vector ANN indices. We'd be using their tooling for a different purpose (program retrieval rather than recommendation), but the substrate is the same.

This trades exactness for tractability — we no longer guarantee we're using the actual top-k by posterior weight, only the top-k programs whose *signatures* match the query well. For most predictions this is fine; for high-stakes predictions you can fall back to exact top-k over a smaller candidate set.

**Self-modifying interpreters at scale.** PLAN.md V13 promises this works in principle. At trillion-program scale, the interpreter is a *family* of versioned programs with shadow-traffic dispatch — every interpreter rewrite gets a small slice of live traffic, performance is measured, version is promoted on win or rolled back on loss. Same operational pattern as web-scale A/B testing, applied to the interpreter rather than UI features.

**The honest open question:** does combinator-driven library growth actually have the favorable scaling we hope for at trillion-program scale? Or does it asymptote because the abstraction operation (V5) eventually finds all the useful primitives and after that we're just adding noise? PLAN.md V20 is the honest report. We genuinely don't know yet.

### Migration path

We don't rewrite at every tier. Migrations are expensive. The realistic path:

- **V1–V8:** stay on the seed design (pickle → LMDB, Modal sharding). Get to 10–100M programs, get the calibration story working, get the BPB numbers respectable. Don't optimize prematurely.
- **V9–V15:** migrate selectively. Move weights and output cache to FDB or TiKV. Add Redis hot tier. Adopt Marisa-trie for n-grams when KenLM hits its limit. Add stratified Bayesian updates.
- **V16–V20:** if BPB numbers justify it, migrate to tier-3 (hierarchical libraries, probabilistic execution, Ray+Kubernetes). If they don't, we have our V20 honest report and we know the architecture has a ceiling. Either outcome is informative.
- **Past V20:** open research. Tier-4 is speculative.

### What this means for the Ralph loop

The Ralph loop reads STORAGE.md as part of its context. When it gets to PLAN.md V8 (Modal sharding), it should reference Tier 2 here for the storage migration. When it gets to V15+ and BPB numbers are pushing past tier-2 capacity, it should reference Tier 3 and ask for human approval before initiating the migration (this is exactly the kind of "major architectural change" that PLAN.md §0 says requires human sign-off).

The Ralph loop should NOT speculatively pre-build tier-3 or tier-4 infrastructure. Premature migration to a tier we don't yet need is the most common way these projects burn budget without producing results. Build tier 1, prove it, migrate to tier 2 when the numbers demand it, and so on.

### Comparison: what an LLM does at the same scale

For honest perspective: a frontier LLM at "trillion parameters" (which is the closest analog to "trillion programs") runs on:
- **Distributed inference clusters** of GPUs interconnected with InfiniBand
- **Tensor parallelism + pipeline parallelism + expert parallelism** sharding the model across ~10K accelerators
- **KV-cache management** to handle long contexts (this alone is a multi-billion-dollar engineering problem at frontier labs)
- **vLLM or TensorRT-LLM** as serving frameworks
- Cost: hundreds of millions of dollars in capex, $100K+ per day in inference compute at production scale

Our tier-3 deployment at ~$100K/mo would be running a "trillion-program" RCE that's structurally different but comparable in expressive capacity (assuming the architecture's central bet pays off). That's the cost asymmetry we're betting on. CPU-bound, no specialized hardware, no exotic networking. If the BPB numbers come out competitive, the cost story alone makes this worth pursuing.

If the BPB numbers don't come out competitive — and that's a real possibility — the calibration story alone may still justify production deployment for high-stakes domains (medical, legal, financial) where "I don't know" is more valuable than fluent confabulation. That's a different product, but the same architecture.
