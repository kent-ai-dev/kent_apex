# RCE — Next Steps for Training and Expanding the System
## Plan v1 → v20, written as an instruction prompt for an LLM agent (Ralph loop)

---

## 0. Read this first (orientation for the executing LLM)

You are the engineer in a Ralph loop. Your job is to evolve the Recurrent
Compression Engine (RCE) toward frontier-model competitiveness through a
sequence of numbered experiments. Each experiment is a self-contained
hypothesis: a change to the system, a measurable predicted outcome, and a
go/no-go criterion before moving to the next version.

**Before any experiment**, read:

1. `engine.py` — the core: programs, combinators, library, Bayesian update
2. `rce.py` — the CLI: train/chat/status/reset
3. `plans/storage.md` — storage architecture, tier system, and migration
   triggers (which versions require switching the persistence layer)
4. `BENCHMARKS.md` — the running record of metrics across versions (you create
   and maintain this; if absent, create it on V1)
5. `LOG.md` — the running journal of every experiment, what changed, what
   happened, what to keep, what to revert (you create and maintain this)

**Architectural philosophy you must respect:**

- The system is a **library of explicit programs** weighted by Bayesian
  posterior with a Solomonoff length prior. Do not replace this with a
  neural net. That is the entire point of the project.
- Growth is by **recurrence (combinators)**, not search. New programs come
  from applying a small fixed set of combinators to existing ones. Do not
  introduce gradient descent over program space.
- **Hallucination must remain structurally impossible.** Every output is a
  sample from an explicit posterior. If you find yourself adding a generator
  that can fabricate, stop and reconsider.
- **Audit trails are sacred.** Every prediction must trace to specific
  weighted programs. Do not introduce opaque components.

**What you may add freely:**

- New primitive program types (more sophisticated predictors)
- New combinators (carefully — each one expands expressible space)
- New tokenizations (bytes are not the only choice)
- New training-data sources (HuggingFace datasets, books, code, etc.)
- Compute scaling (Modal for distributed training)
- Better posterior management (regularization, decay, replay)
- Evaluation harnesses (perplexity, downstream tasks, calibration tests)

**Mandatory pre-ingestion check.** Before any new HuggingFace dataset is
downloaded for training, run `python validate_dataset.py <name> --config
<config>`. The validator runs a streaming-load + iteration + schema +
text-field + UTF-8 check and emits a 0–100 "vibes" score. Verdicts:
PASS (≥85) proceeds silently; WARN (60–84) proceeds but the operator
must note the warning in `LOG.md`; FAIL (<60) blocks ingestion — the
operator picks an alternate dataset, validates *that*, and only then
proceeds. Dealing with corrupt data after compute has been spent is the
expensive failure mode this gate prevents.

**What you must justify in writing before adding:**

- Anything that breaks the audit-trail property
- Anything that introduces a non-Bayesian update rule
- Anything that requires a neural network
- Anything that increases library size by more than 10× in one step

## 1. How this is different from an LLM (and why the experiments target what they target)

| Axis | Frontier LLM | RCE |
|------|-------------|-----|
| Representation | Continuous tensor (~10¹² params) | Discrete program library |
| Training | Gradient descent on cross-entropy | Bayesian update + combinator generation |
| Inference | Single forward pass | Weighted vote across programs |
| Knowledge | Implicit, distributed in weights | Explicit, traceable to programs |
| Uncertainty | Logit calibration (unreliable) | Posterior entropy (computed exactly) |
| Hallucination | Emergent | Structurally impossible |
| Self-modification | None (fixed architecture) | Interpreter is a program; combinators can recur on themselves |
| Compute profile | Massive parallel matmul | Many small parallel program evaluations |
| Failure mode | Confident wrong answers | Refuses to answer (high entropy) |

The experimental program below is designed to attack each axis where the
RCE is currently weaker than an LLM (representational capacity, training
throughput, evaluation rigor) without sacrificing the axes where it is
stronger (auditability, calibration, self-modification).

## 2. Training philosophy (read carefully — this is NOT LLM training)

You are not doing gradient descent. You are not minimizing a scalar loss
across a held-out set with backprop. The training loop is:

1. **Stream bytes** (or tokens) from a corpus
2. **Each program in the library predicts** the next byte from recent context
3. **Bayesian posterior update**: programs whose distributions assigned
   higher probability to the actual byte gain log-weight; others lose
4. **Periodically: grow** — apply combinators to top-weighted programs to
   produce children
5. **Periodically: prune** — drop low-weight non-primitive programs
6. **Periodically: abstract** — when a sub-program appears in many
   high-weight programs, lift it to a new primitive (DreamCoder-style
   wake-sleep, but pure-symbolic)

There is no validation loss, no learning rate schedule, no warmup, no
checkpoint averaging. There is the posterior. The posterior IS the
training state. The library snapshot IS the model.

**Datasets:** HuggingFace datasets are perfect inputs. They are byte
streams (after tokenization or raw). Use `datasets` library to stream them.
Do NOT pre-shuffle into batches the way an LLM trainer does — that destroys
sequential structure the RCE needs. Stream in document-order.

Recommended HF datasets, by phase:

- **V1–V5 (toy):**
  [wikitext-2-raw-v1](https://huggingface.co/datasets/Salesforce/wikitext),
  [tiny_shakespeare](https://huggingface.co/datasets/karpathy/tiny_shakespeare),
  [ag_news](https://huggingface.co/datasets/fancyzhx/ag_news)
- **V6–V10 (small):**
  [wikitext-103-raw-v1](https://huggingface.co/datasets/Salesforce/wikitext),
  [bookcorpus](https://huggingface.co/datasets/bookcorpus/bookcorpus) subset,
  [pile-uncopyrighted](https://huggingface.co/datasets/monology/pile-uncopyrighted) subset (~1GB)
- **V11–V15 (medium):**
  [SlimPajama-6B](https://huggingface.co/datasets/DKYoon/SlimPajama-6B),
  [c4](https://huggingface.co/datasets/allenai/c4) English subset,
  [code_search_net](https://huggingface.co/datasets/code-search-net/code_search_net)
- **V16–V20 (large):**
  [pile-uncopyrighted](https://huggingface.co/datasets/monology/pile-uncopyrighted) full,
  [RedPajama-Data-1T](https://huggingface.co/datasets/togethercomputer/RedPajama-Data-1T),
  [dolma](https://huggingface.co/datasets/allenai/dolma)

**Compute:** For V1–V8, a single laptop is fine. For V9+, use Modal:
- Modal lets you run Python functions on GPUs/CPUs in the cloud with
  zero infrastructure setup.
- The RCE does not need GPUs (no matmul). It needs many CPUs running
  many program evaluations in parallel.
- Modal's `@app.function(cpu=8, memory=16384)` decorator gives you that.
- For distributed training: use Modal's `.map()` to shard the corpus
  across workers, each maintaining a sub-library, then merge libraries
  on a master node by union + posterior renormalization.
- Custom domains (modal.com/docs/guide/webhook-urls#custom-domains) are
  useful in V18+ when exposing the chat endpoint as a service.

## 3. Plans V1 — V20 (in order; do not skip)

Each plan has: **goal**, **change**, **measure**, **gate** (must pass
before next version).

### V1 — Establish the benchmark harness
**Goal:** Be able to measure progress.
**Change:** Add `bench.py` with three metrics:
1. **Bits-per-byte (BPB)** on a held-out 100KB slice of `wikitext-2-raw-v1`.
   Compute: `-mean(log2(P(actual_byte | ctx)))` across the slice.
2. **Calibration AUC**: bin predictions by confidence, measure accuracy
   per bin, compute ECE (Expected Calibration Error).
3. **Refusal rate** on out-of-distribution inputs (random binary, foreign
   scripts not in training): fraction where entropy > 7.0.
**Measure:** Record baseline numbers for current system on `wikitext-2`.
**Gate:** Numbers reproducible across two runs within ±2%.
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (config `wikitext-2-raw-v1`) — small enough for fast iteration on the harness itself.

### V2 — Tokenization sweep
**Goal:** Bytes are not the only input. Try alternatives.
**Change:** Add a `Tokenizer` abstraction. Implement: bytes (current),
BPE (use `tokenizers` library, train a 4K vocab), word-level (split on
whitespace + punctuation).
**Measure:** BPB across all three tokenizations on `wikitext-2`.
**Gate:** Pick the tokenization with best BPB. Document why. (Hypothesis:
BPE will win because it shortens effective context and fits more pattern
in n-gram tables.)
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (`wikitext-2-raw-v1`) — same corpus as V1 so the BPB delta isolates tokenization.

### V3 — Richer primitive predictors
**Goal:** Current primitives are uniform, repeat, n-grams. Add stronger ones.
**Change:** Implement these primitives:
- `SuffixArrayPredictor`: full suffix array of training data; predicts by
  longest matching suffix (Markov of unbounded order). This is the
  Prediction-by-Partial-Match (PPM) family — known to be strong on text.
- `LZWPredictor`: build LZW dictionary; predict by dictionary entry probabilities.
- `KneserNeyNGram(n)`: n-gram with Kneser-Ney smoothing instead of add-k.
- `SkipGramPredictor(k)`: predict from non-adjacent context (e.g., bytes
  at offset -1, -3, -5).
**Measure:** BPB delta vs V2.
**Gate:** At least one new primitive beats best n-gram by >5% BPB.
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (`wikitext-2-raw-v1`) — hold V1/V2 corpus constant; only the predictor set changes.

**Hazard note (refusal regression — read before implementing):**
KneserNeyNGram's order-0 base case is the *continuation distribution*
(how many distinct contexts each byte appears as a continuation in).
On OOD input where higher orders all miss in the backoff, KN returns
this learned prior — entropy ≈5 bits, well below V1's 7.0 refusal
threshold. The library's posterior-weighted prediction inherits this
confidence and the OOD refusal heuristic silently breaks (V1 refusal
1.0000 → V3 refusal 0.0000 on `os.urandom` is the predicted
regression).

This is the architecture's foundational "hallucination structurally
impossible" claim breaking at the V3 commit. Do **not** paper over
it. Two prohibited shortcuts:
- **Don't** patch by clamping KN's order-0 to uniform — it loses
  calibration info on legitimate low-context positions and hides the
  problem rather than fixing it.
- **Don't** raise the refusal threshold above 7.0 — that's
  threshold-tuning, not an architectural fix, and breaks the moment
  any other primitive learns a prior.

The correct fix lands at **V10 (Calibration as a feature)** as a
top-level Bayesian mixture with an explicit Background model. V10's
refusal invariant must restore `os.urandom` refusal to ≥0.95 *while
KN remains in the library*. Until V10 ships, V4-V9 carry a known
broken architectural invariant — log it in `LOG.md` and `state.json`
under `open_findings`, accept that intermediate BPB gates are still
informative for the predictor work but the calibration story is on
hold, and do not promise the V20 honest report will land without
V10's fix.

### V4 — Combinator expansion
**Goal:** Four combinators is the minimum. Try more.
**Change:** Add three new combinators:
- `Gate(a, b, predicate)`: use `a` if a learned predicate fires on
  context, else `b`. Predicate is itself a small program.
- `Memo(a)`: cache `a`'s predictions on recently-seen contexts.
- `Mixture(a, b, c, ...)`: posterior-weighted mixture of >2 parents
  (current `Composed` is binary).
**Measure:** Library composition after training. Count how many
high-weight programs use each combinator type.
**Gate:** At least one new combinator appears in top-20 programs after
training. If not, the combinator is dead weight — remove it and document why.
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (`wikitext-2-raw-v1`) — same corpus; library composition is the dependent variable.

### V5 — Wake-sleep abstraction
**Goal:** When a sub-program appears in many high-weight programs, lift
it to a new primitive. This is the engine of language growth.
**Change:** Add `abstract_phase()`: scan top-50 programs for shared
sub-trees in their combinator lineage; if a sub-tree appears ≥3 times,
create a new primitive that directly computes it (caching predictions),
add to library, and replace occurrences in parent programs with
references to the new primitive.
**Measure:** Library size over training time. With abstraction, library
size should asymptote even as data grows (compression).
**Gate:** Library asymptotes within 2× the size of V4 on the same data.
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (`wikitext-2-raw-v1`) — repeated patterns in news/wiki prose make this a fair test for abstraction lifting.

### V6 — Streaming HuggingFace ingestion
**Goal:** Move from local files to real datasets.
**Change:** Add `datasets/hf.py`. Implement streaming ingest from any
HF dataset by name. Use `datasets.load_dataset(name, streaming=True)`
and iterate. Add CLI: `python rce.py train --hf wikitext-103-raw-v1
--max-bytes 10_000_000`.
**Measure:** Throughput in bytes/second on a single laptop core.
**Gate:** Sustained ≥50KB/s ingestion without library blowup.
**Dataset:** [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) (`wikitext-103-raw-v1`) — 50× larger than V1's slice; first real test that the streaming path holds up.
**Storage migration (per `plans/storage.md`):** the single-pickle store breaks down somewhere around 100K programs, and V6 is when datasets get big enough to push us there. Migrate weights and output cache to **LMDB**; keep program code as a directory of pickles. Write `migrate_v6.py` that reads the seed pickle and emits the LMDB structure. Verify post-migration BPB matches pre-migration BPB to within numerical noise (deterministic round-trip).

### V7 — Posterior decay and replay
**Goal:** Stationary Bayesian posteriors over-commit on early data.
The brain solves this with sleep-replay and synaptic homeostasis.
**Change:** Two additions:
1. **Decay**: every N bytes, multiply all log-weights by 0.99. Keeps
   posterior responsive to non-stationary data.
2. **Replay buffer**: sample 1K random past bytes per growth phase,
   re-run posterior updates on them. Prevents catastrophic forgetting.
**Measure:** Train on corpus A, then corpus B; measure BPB on A held-out.
With/without replay.
**Gate:** Replay version retains ≥80% of A's BPB after training on B.
**Dataset:** corpus A = [karpathy/tiny_shakespeare](https://huggingface.co/datasets/karpathy/tiny_shakespeare); corpus B = [fancyzhx/ag_news](https://huggingface.co/datasets/fancyzhx/ag_news) — two sharply different distributions (Early Modern English vs. modern news headlines) maximize the catastrophic-forgetting signal.

### V8 — Parallel training on Modal
**Goal:** Single-laptop training won't scale to interesting datasets.
**Change:** Set up Modal app. Sharding strategy:
- Split corpus into N shards (one per worker).
- Each worker maintains its own library, trains independently.
- Master process periodically merges libraries: union of programs,
  log-weights summed where same program appears, then renormalize.
- Use Modal's volume mounts to persist libraries between runs.
**Measure:** Wall-clock time to process 100MB corpus on 1, 4, 16 workers.
**Gate:** Linear scaling within 30% (sub-linear is fine — true linear
would be miraculous given merge costs).
**Dataset:** [bookcorpus/bookcorpus](https://huggingface.co/datasets/bookcorpus/bookcorpus) — 100MB sliceable subset of long-form prose; document-order shards naturally without breaking sentence/paragraph structure.
**Storage migration (per `plans/storage.md` Tier 1):** add the master/worker merge protocol — workers train independently on weight shards (LMDB-per-worker), master sums log-weights and renormalizes every N bytes. Replace n-gram primitives with **KenLM** mmap'd indices so n-gram primitives don't have to be re-fitted per worker. Update math: when merging M workers, subtract `(M-1) × Solomonoff_prior` per program to avoid double-counting the prior (one of the issues flagged in v1.md review).

### V9 — First serious training run
**Goal:** Train at scale and see what happens.
**Change:** On Modal with 32 workers, train on 1GB of `the_pile`
(English subset). Use V2's best tokenization, V3+V4 primitives, V5
abstraction, V7 decay+replay.
**Measure:**
- Final library size
- BPB on held-out wikitext-103
- Top-20 programs by weight (log them all)
- Time-to-train
**Gate:** BPB ≤ 2.5 bits per byte on wikitext-103 held-out. (For
context: a small LSTM gets ~1.5; raw bytes are 8; the goal here is to
beat naive n-gram baselines decisively.)
**Dataset:** train on [monology/pile-uncopyrighted](https://huggingface.co/datasets/monology/pile-uncopyrighted) (1GB English subset; the_pile mirror that survived the EleutherAI takedown); evaluate on held-out [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) `wikitext-103-raw-v1`.

### V10 — Calibration as a feature (and the architectural refusal fix)
**Goal:** Two coupled goals.

  (a) **Repair the V3 refusal regression at the architectural level.**
      The "hallucination structurally impossible" claim is the single
      most-marketed property of the system and is currently false in
      code. Every benchmark gain from V4-V9 inherits that lie. Until
      refusal is restored, the V20 honest report cannot honestly say
      the architecture's foundational claim holds. Treat this as a
      non-negotiable architectural fix, not an optional polish step.

  (b) Make honest uncertainty *measurable and useful*. The RCE's
      biggest structural advantage over LLMs is calibrated refusal;
      build the dashboard and modes that surface it.

**Change:** Four sub-changes, in this order. **0 is blocking on
1-3.** Do not advance to 1-3 until 0's gates have passed.

0. **Refusal as a structural invariant (priority 0).** Replace the
   entropy-threshold refusal heuristic with an explicit Bayesian
   mixture at the *top level* of the library, between the trained
   library and a literal Background model that returns uniform
   over the vocabulary.

   ```python
   # engine.py — new class
   class Toplevel(Program):
       """
       Bayesian mixture over (in-distribution library, background uniform).
       The architectural refusal invariant lives here.
       """
       def __init__(self, in_dist: Library, vocab_size: int):
           self.in_dist     = in_dist
           self.background  = UniformPrimitive(vocab_size)
           # log posterior weights — initialised at the Solomonoff prior
           # (background is shorter to describe; gets a small prior bonus)
           self.log_w_in    = 0.0
           self.log_w_bg    = -log2(vocab_size_complexity_bits) or 0.0

       def predict(self, ctx) -> Counter[token]:
           p_in = self.in_dist.predict(ctx)          # library mixture
           p_bg = self.background.predict(ctx)       # uniform over vocab
           w_in, w_bg = softmax([self.log_w_in, self.log_w_bg])
           return mix(p_in, p_bg, w_in, w_bg)

       def update(self, ctx, actual_token):
           # standard Bayesian: log_weight += log P(actual | model)
           p_in_act = self.in_dist.predict(ctx)[actual_token]
           p_bg_act = 1.0 / self.background.vocab_size
           self.log_w_in += log(max(p_in_act, EPS))
           self.log_w_bg += log(p_bg_act)
           # also propagate the update into self.in_dist's
           # per-program posterior (existing behaviour)
           self.in_dist.update(ctx, actual_token)

       def refusal_score(self, ctx) -> float:
           """P(Background | ctx) — the refusal probability."""
           p_in = self.in_dist.predict(ctx)
           p_bg = self.background.predict(ctx)
           # marginal likelihood of ctx under each branch * prior
           # (Bayes rule; computed via log-sum-exp for stability)
           return posterior_on_background(
               p_in, p_bg, self.log_w_in, self.log_w_bg
           )
   ```

   **Why this works where the entropy heuristic failed.** KN's
   confident-but-wrong predictions on OOD have low likelihood on the
   actual byte (which is genuinely random or foreign-script).
   Background uniformly assigns `1/|vocab|` to every byte. So on OOD,
   Background's likelihood ratio steadily grows over a window;
   on in-distribution, in_dist dominates. Refusal is a measured
   posterior probability, not a hand-tuned threshold. Adding new
   smoothing primitives in the future cannot silently break this
   property — they can only change *how fast* Background catches up
   on OOD, which is observable in the benchmark numbers.

   **Implementation steps (small commits, in order):**

   - **i.** Add `UniformPrimitive(vocab_size)` if not present
     (`engine.py`). Already exists for byte vocab; verify it works
     for word vocab (V2 winner) and any V11+ tokenizations.
   - **ii.** Add `Toplevel(Program)` class to `engine.py` with the
     interface above. Keep it in the same file so `save_library` /
     `load_library` round-trip naturally.
   - **iii.** Modify `fresh_library()` to return
     `Toplevel(Library(...), vocab_size)` instead of bare `Library`.
     This is a breaking change to the public interface; bump the
     library pickle/LMDB schema version and write a one-shot
     migrator (`migrate_v10.py`) that wraps existing snapshots.
   - **iv.** Update `bayes_train` so each step calls
     `toplevel.update(ctx, actual)` — which transparently updates
     both the toplevel mixture and the inner library's per-program
     posterior. No separate update path.
   - **v.** Update `rce.py` chat to consult `toplevel.refusal_score`
     before emitting; if `score > τ` (default `τ = 0.5`), emit a
     refusal token and surface the explanation (item 3 below).
   - **vi.** Update `bench.py`: refusal metric is now
     `mean(toplevel.refusal_score(ctx) > τ)` over the OOD slice,
     replacing the entropy-threshold computation. The old metric
     stays as a secondary diagnostic so we can compare regimes.
   - **vii.** Audit-trail wiring: every refusal decision logs the
     in-distribution likelihood, background likelihood, current
     posterior split, and the top-3 contributing programs from
     in_dist. Surfaced through `--explain` (item 3).

   **Cross-version regression tests (must pass before advancing):**

   - Re-run V1's bench harness configuration on a *clean* (untrained)
     library wrapped in `Toplevel`. Refusal on `os.urandom` must
     return to ≥0.99 with `τ = 0.5`. This tests that the wrapper
     doesn't break the original property by construction.
   - Re-run V3's KN-trained configuration *with* `Toplevel`. Refusal
     on `os.urandom` must climb back from 0.0 (V3) to ≥0.95. This
     is the actual fix: KN remains in the library; the architectural
     mixture absorbs its OOD overconfidence.
   - Re-run V7's decay+replay configuration with `Toplevel`. Refusal
     on the `corpus A → corpus B` shift must show the Background
     posterior reactivating during the shift transient (this is the
     *expected* behaviour and is itself a useful calibration signal).
   - Bench `Toplevel` on in-distribution wikitext-103: BPB cost of
     adding the mixture must be ≤5% over V9's number. (The Background
     pulls a small amount of likelihood on every prediction; quantify
     it. If >5%, raise `log_w_bg` initial penalty until it sits within
     budget without changing the OOD refusal numbers.)

   **What NOT to do (prohibited shortcuts):**
   - Don't clamp KN's order-0 base case to uniform. Hides the
     problem in one primitive and leaves every future primitive at
     risk of the same regression.
   - Don't tune the V1 entropy threshold upward. That's calibration
     theatre, not an invariant.
   - Don't make the Background a *trained* model. Background must be
     uniform-over-vocab by construction; otherwise it cannot serve
     as the architectural reference point that "refuse" means.
   - Don't ship 1-3 below until 0's gates pass. The dashboard and
     strict mode are useless if refusal itself is broken.

1. Build a calibration dashboard: for each prediction during a held-out
   evaluation, plot stated confidence vs actual accuracy. Confidence is
   computed against the **Toplevel** prediction (which already weights
   in_dist against Background), not against the bare inner library.
   The dashboard surfaces three curves side-by-side: in_dist confidence,
   toplevel confidence, refusal_score — so the calibration story is
   visible per-input.

2. Add a `--strict` chat mode: refuse to generate any byte where
   *either* `toplevel.refusal_score > 0.5` *or* the toplevel top
   prediction's confidence < 0.3. Strict mode is now governed by the
   architectural invariant from priority 0, not a hand-tuned cutoff.

3. Add a `--explain` chat mode: after each generated byte (or
   refusal), print:
   - The top-3 programs from in_dist that voted for the chosen
     byte, with their posterior weights
   - The toplevel posterior split (`P(in_dist | ctx)` vs
     `P(Background | ctx)`)
   - On refusal: a one-line reason — e.g. "refused: P(Background) =
     0.83 over the last 32 ctx steps; in_dist top-3 voted for byte
     X with weights {…} but Background's uniform likelihood on the
     unseen actual byte outweighed them."

**Measure:**
- Refusal rate on `os.urandom` (V1 baseline 1.0; V3 regression 0.0;
  V10 target ≥0.95 with KN still in the library)
- Refusal rate on Cyrillic text (foreign-script OOD; new metric at
  V10; target ≥0.90)
- ECE on in-distribution wikitext-103 (≤0.05)
- Strict-mode coverage vs accuracy on the answered subset
- BPB on wikitext-103 — must not regress more than 5% vs V9, since
  the toplevel mixture pulls a small amount of likelihood on every
  prediction. Quantify and document the cost.

**Gate (all must pass — priority 0 is blocking on 1-3):**
- Refusal ≥0.95 on `os.urandom` (V3 regression repaired)
- Refusal ≥0.90 on Cyrillic (new OOD axis holds)
- ECE ≤0.05 on wikitext-103
- BPB on wikitext-103 within 5% of V9's number
- Strict mode answers ≥30% of prompts with ≥90% byte-level accuracy

**Dataset:** held-out slice of [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) `wikitext-103-raw-v1` for in-distribution; existing 20KB `os.urandom` for noise OOD; new 20KB Cyrillic slice from [wikimedia/wikipedia](https://huggingface.co/datasets/wikimedia/wikipedia) ru-config for script OOD. Run the new Cyrillic slice through `validate_dataset.py` before any compute (per §0's mandatory pre-ingestion check).

**Storage migration (per `plans/storage.md`):** the toplevel mixture
adds two scalars (`log_w_in`, `log_w_bg`) per saved library. Trivial
schema bump in the LMDB weight store; the one-shot `migrate_v10.py`
wraps existing snapshots in `Toplevel(...)` with default initial
weights. No new tier required.

**Why V10 is the architectural lynchpin:** the V3 refusal regression
went from 1.0000 to 0.0000 silently. Every version V4-V9 inherits the
break. The "hallucination structurally impossible" claim is the most
distinctive property of the architecture and is currently a lie at
the level of the actual codebase. Until V10 lands, every BPB gate
already passed is questionable: any in-distribution win bought by
smoothing primitives that confidently confabulate on OOD is not
progress on the architecture's stated goals. **V10 is a blocking gate
for V20's honest report. Treat it as such.**

### V11 — Token-level evaluation against LLM baselines
**Goal:** Be honest about where we stand.
**Change:** Run the RCE on standard LLM benchmarks, treating it as a
language model:
- **LAMBADA**: predict last word of a passage
- **HellaSwag**: multiple-choice continuation (score each option's
  posterior probability under the library)
- **WikiText-103 perplexity**
- **TriviaQA** (closed-book): hopeless, but worth recording the
  refusal rate
**Measure:** Score on each. Compare to: random, n-gram baseline, GPT-2
small, GPT-2 large.
**Gate:** Beat n-gram baseline on ≥3 of 4 benchmarks. Do NOT expect to
beat GPT-2 yet — record the gap and the gap's slope across versions.
**Datasets:**
[EleutherAI/lambada_openai](https://huggingface.co/datasets/EleutherAI/lambada_openai) (LAMBADA, the eval-harness standard variant);
[Rowan/hellaswag](https://huggingface.co/datasets/Rowan/hellaswag) (HellaSwag);
[Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext) `wikitext-103-raw-v1` (perplexity);
[mandarjoshi/trivia_qa](https://huggingface.co/datasets/mandarjoshi/trivia_qa) (TriviaQA).

### V12 — Cross-domain training
**Goal:** A library trained on text-only is brittle. Diversify.
**Change:** Train on a mixture: 50% text (`c4`), 25% code
(`code_search_net`), 25% structured (`wikidata` triples). Use a
domain-tag prefix byte so combinators can branch on domain.
**Measure:** BPB per domain on held-out. Library composition by
domain (which programs activate where).
**Gate:** Per-domain BPB no worse than 1.2× the single-domain BPB.
(Multi-domain training shouldn't catastrophically harm any one domain.)
**Datasets:**
text 50% — [allenai/c4](https://huggingface.co/datasets/allenai/c4) (English subset, streaming);
code 25% — [code-search-net/code_search_net](https://huggingface.co/datasets/code-search-net/code_search_net);
structured 25% — [intfloat/wikidata5m](https://huggingface.co/datasets/intfloat/wikidata5m) (Wikidata triples; substitute for the abandoned plain-`wikidata` mention).

### V13 — Self-modifying interpreter
**Goal:** The architecture promises self-modification. Deliver it.
**Change:** Move the interpreter (the predict() dispatch logic) into
the program library itself, written in the same combinator language.
Allow combinators to apply to interpreter-programs, producing
candidate replacement interpreters. Run candidates in sandbox; adopt
if they reduce description length of the library OR speed up
inference without harming BPB.
**Measure:** Number of interpreter rewrites accepted across a long
training run. BPB and inference speed before/after.
**Gate:** At least one interpreter rewrite is accepted. (This is the
Gödel Machine moment — even one accepted self-rewrite validates the
architecture's central promise.)
**Dataset:** reuse V12's mixture so BPB-before/after is comparable; if a single dataset is needed, [DKYoon/SlimPajama-6B](https://huggingface.co/datasets/DKYoon/SlimPajama-6B) provides clean diverse text without re-introducing domain shift.
**Storage migration (per `plans/storage.md`):** the interpreter is a stored program now. Storage must support **copy-on-write** semantics — candidate interpreter rewrites run against a CoW snapshot of the library, and atomic-swap on accept. LMDB transactions cover this for ≤100M-program libraries; document the rollback path explicitly (failed rewrite → discard CoW snapshot, no state change).

### V14 — Active curiosity (intrinsic motivation)
**Goal:** Currently the system passively ingests whatever you feed it.
Add Schmidhuber-style curiosity: prefer training data that maximally
reduces prediction error.
**Change:** When connected to a streaming source with selectable
subsets, the trainer scores incoming chunks by **prediction error
under current library**. High-error chunks get prioritized in the
training queue. (This is the compression-progress drive.)
**Measure:** With curiosity on/off, learning rate (BPB drop per byte
of training).
**Gate:** Curiosity-on trains to target BPB in ≤70% the bytes of
curiosity-off.
**Dataset:** [DKYoon/SlimPajama-6B](https://huggingface.co/datasets/DKYoon/SlimPajama-6B) — pre-deduplicated, multi-source, streamable; the natural source-mixture variance gives curiosity meaningful selectivity.

### V15 — Conversational fine-grain
**Goal:** RCE chat output is currently noisy. Make it conversational.
**Change:**
1. Train on conversational corpora: `daily_dialog`, `persona_chat`,
   `oasst1`.
2. Add a `<dialog>` framing primitive that learns turn structure.
3. Generation: stop on turn-boundary tokens (newline + role marker).
4. Add `--continue` mode that maintains chat context across turns
   in the input.
**Measure:** Human evaluation on 50 prompts: rate coherence 1–5.
**Gate:** Mean coherence ≥3.0 (recognizable as on-topic, even if not
fluent).
**Datasets:**
[li2017dailydialog/daily_dialog](https://huggingface.co/datasets/li2017dailydialog/daily_dialog) (everyday two-party dialog with turn structure);
[bavard/personachat_truecased](https://huggingface.co/datasets/bavard/personachat_truecased) (PersonaChat — persona-conditioned dialogue);
[OpenAssistant/oasst1](https://huggingface.co/datasets/OpenAssistant/oasst1) (assistant-style multi-turn).

### V16 — Knowledge persistence and provenance
**Goal:** Use the audit-trail advantage. Every fact the system "knows"
should be traceable.
**Change:** When a program achieves high posterior weight, log:
- The training shard(s) that drove its weight up
- The contexts where it consistently predicts well
- A "domain signature" (which byte n-grams it activates on)
Surface this in `status` and via a new `provenance <query>` command
that says: "When asked about X, programs P₁, P₂, P₃ activate, and
they were primarily strengthened by data from sources S₁, S₂."
**Measure:** Coverage: what fraction of high-weight programs have
non-trivial provenance.
**Gate:** ≥80% of top-100 programs have meaningful provenance traces.
**Dataset:** the V12 mixture, plus a held-out evaluation slice from [allenai/dolma](https://huggingface.co/datasets/allenai/dolma) — Dolma's documented per-source manifest is what makes provenance verifiable end-to-end (every byte ingested has a known origin).
**Storage migration (per `plans/storage.md`):** add a dedicated **append-only provenance store** (one JSONL file per program ID, never deleted even when the program is pruned). Provenance entries record: training shards that drove weight up, contexts where the program predicts well, domain signature. Persist alongside the LMDB stores; never inside them, since we want provenance to survive pruning of the program itself.

### V17 — Distributed library federation
**Goal:** Why train one library when you can federate many?
**Change:** Build a library exchange format (gzipped pickle + signed
manifest of provenance). Two RCE instances can exchange libraries by
union, with weights renormalized and provenance preserved. Train
specialist libraries (one on code, one on math, one on dialogue),
then federate.
**Measure:** Federated library BPB on each domain vs specialist BPB
on its own domain.
**Gate:** Federation loses ≤15% BPB per domain vs specialist. (The
bet: combinator generation will discover cross-domain abstractions
that specialists couldn't.)
**Datasets:** specialist libraries trained on
[code-search-net/code_search_net](https://huggingface.co/datasets/code-search-net/code_search_net) (code),
[OpenAssistant/oasst1](https://huggingface.co/datasets/OpenAssistant/oasst1) (dialogue),
[intfloat/wikidata5m](https://huggingface.co/datasets/intfloat/wikidata5m) (structured triples) — three sharply different distributions stress-test the merge math.

### V18 — Public chat endpoint via Modal
**Goal:** Expose to real users; gather real prediction-error signal.
**Change:** Modal web endpoint with custom domain. The chat endpoint
streams responses. Every conversation is logged with the user's
ratings (thumbs up/down per turn). Ratings feed back into curiosity
(V14) — high-disagreement turns become training-priority data.
**Measure:** Daily active conversations. Rating distribution.
Library growth from real-user data.
**Gate:** Sustained ≥10 conversations/day for one week. Library
grows ≥10% from user-generated curiosity signal.
**Dataset:** real users — but warm-start the deployed library on [OpenAssistant/oasst1](https://huggingface.co/datasets/OpenAssistant/oasst1) so first-turn coherence is high enough that users actually return. User logs become the live training signal from there.

### V19 — Multi-modal extension
**Goal:** Bytes are universal. Try non-text.
**Change:** Train on raw image bytes (small images, e.g., MNIST as
flat byte streams) and audio (8-bit μ-law samples). Same combinators,
same library.
**Measure:** BPB on images and audio. Visualize what programs the
library evolves for non-text data — do recur(recur(ngram)) chains
discover edge detection? Compositional structure?
**Gate:** BPB on images ≤ 6.5 (vs 8 for raw, vs ~4 for PNG).
This is a stretch goal; failure here is fine and informative.
**Datasets:** images — [ylecun/mnist](https://huggingface.co/datasets/ylecun/mnist) (28×28 grayscale, ideal as flat byte streams); audio — [openslr/librispeech_asr](https://huggingface.co/datasets/openslr/librispeech_asr) (use `clean-100` config and downsample to 8-bit μ-law).

### V20 — The honest report
**Goal:** Write up where we are. Be ruthlessly honest.
**Change:** Compose `RESULTS.md`:
- Final BPB on each benchmark
- Calibration metrics (where we likely beat LLMs)
- Cost per byte trained vs LLM equivalents
- Honest assessment of frontier-competitiveness
- What worked, what didn't, what's next
**Measure:** N/A — this is the report.
**Gate:** Document is publishable as a research note.
**Dataset:** no new ingest — the report consolidates results across every prior version's dataset. Cite the HuggingFace pages directly so readers can reproduce.

## 4. The Ralph loop (continuous trainer/director)

The Ralph loop is a long-running script that:
1. Reads the current version state from `state.json`
2. Loads this plan
3. Determines the next experiment to run
4. Sends instructions to a coding-capable LLM (Claude, via API)
5. Receives code/commits back, applies them
6. Runs the experiment
7. Logs results
8. Decides: gate passed → next version; gate failed → diagnose, retry,
   or escalate to human
9. Sleep, loop

The Ralph loop's prompt to the LLM each iteration is the next section.

---

## 5. The Ralph-loop prompt (this is what gets sent to the LLM each iteration)

```
You are Claude, acting as the engineer in a Ralph loop training the RCE.

CURRENT STATE:
- Version: {version}
- Last gate result: {pass|fail|in_progress}
- Last commit: {sha}
- Time since last human review: {hours}h

YOUR TASK THIS ITERATION:
Read these files in order:
  1. /workspace/rce/PLAN.md (this plan)
  2. /workspace/rce/LOG.md (running journal)
  3. /workspace/rce/BENCHMARKS.md (running metrics)
  4. /workspace/rce/engine.py
  5. /workspace/rce/rce.py
  6. Any V{version}-related files already started

Then determine:
  (a) What stage of V{version} we are at:
      - DESIGN: no code written for this version yet
      - IMPLEMENT: code partially written, not running
      - RUN: code complete, experiment in progress
      - EVALUATE: experiment complete, gate not yet checked
      - GATE_FAIL: gate failed, need to diagnose or retry
      - DONE: gate passed; advance to V{version+1}

  (b) The single next concrete action.

CONSTRAINTS:
  - Make ONE focused change per iteration. No mass rewrites.
  - Every change must be auditable: write to LOG.md what you did and why.
  - Every metric must go to BENCHMARKS.md as a row with date+version+score.
  - You MAY NOT introduce: neural networks, gradient descent,
    non-Bayesian update rules, or anything that breaks audit trails.
    If you believe an exception is warranted, write it in LOG.md as a
    PROPOSAL and stop. Do not implement until a human approves.
  - You MAY introduce: new primitives, combinators, tokenizations,
    datasets, evaluation harnesses, infrastructure (Modal), provenance
    tooling, calibration tooling.
  - If a gate fails twice with different fixes, escalate: write
    ESCALATE in LOG.md with diagnostic notes and stop.

OUTPUT FORMAT:
  1. STATE_ASSESSMENT: which stage you're in, what you read, what you found
  2. NEXT_ACTION: one specific change (code edit, command to run, file
     to create), with the exact diff or command
  3. EXPECTED_OUTCOME: what you expect to see after this action
  4. UPDATE_LOG: the line(s) to append to LOG.md
  5. UPDATE_BENCHMARKS: the row(s) (if any) to append to BENCHMARKS.md

Then perform the action via available tools. Then exit.

The Ralph loop will sleep, then call you again with updated state.
```

## 6. Bootstrapping the Ralph loop

To launch:

```bash
# Initial setup (human runs once, from the repo root)
mkdir -p /workspace/rce
cp engine.py rce.py /workspace/rce/
cp plans/v1.md /workspace/rce/PLAN.md
cd /workspace/rce
printf '{"version": 1, "stage": "DESIGN", "iteration": 0, "last_gate": "in_progress"}\n' > state.json
touch LOG.md BENCHMARKS.md

# Install the only external dep
pip install -r requirements.txt   # see repo root

# Launch the loop (from repo root, pointing at the workspace)
export ANTHROPIC_API_KEY=sk-ant-...
python ralph.py --workspace /workspace/rce --interval 30m --max-iters 1000

# Or sanity-check the prompt without spending tokens:
python ralph.py --workspace /workspace/rce --dry-run
```

`ralph.py` is a thin orchestrator: it calls Claude with the prompt above,
applies the returned actions, runs commands, captures outputs, updates
state, sleeps, repeats. It does not contain RCE logic. It is the
conductor, not the orchestra.

Reasonable safety rails for `ralph.py`:
  - Hard timeout per iteration (e.g., 60 min)
  - Disk quota check before each train run
  - Modal cost ceiling check before each cloud experiment
  - Required human review every 10 iterations OR on any ESCALATE flag
  - Append-only LOG.md (never rewrites history)

## 7. What success looks like at V20

You will have:
- A library of ~100K–1M small programs
- BPB on wikitext-103 between 2.0 and 2.5
- Calibration ECE ≤ 0.05 (likely better than any LLM, full stop)
- Honest refusal rate >40% on out-of-distribution inputs
- Demonstrable self-modification (interpreter rewrites accepted)
- Public chat endpoint with real users
- Compute cost orders of magnitude below an LLM of equivalent BPB
- A complete audit trail for every prediction

What you will NOT have:
- Frontier-LLM-level fluency on open-ended generation
- Strong performance on tasks requiring world knowledge the library
  hasn't explicitly compressed
- Anything resembling reasoning ability beyond what the combinator
  graph naturally produces

The plan is designed to test the central architectural bet: that
combinator-driven library growth is a real alternative scaling law,
not just a clever-looking dead end. By V20 we will know.
