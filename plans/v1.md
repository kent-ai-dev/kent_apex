# RCE — Next Steps for Training and Expanding the System
## Plan v1 → v20, written as an instruction prompt for an LLM agent (Ralph loop)

---

## 0. Read this first (orientation for the executing LLM)

You are the engineer in a Ralph loop. Your job is to evolve the Recurrent
Compression Engine (RCE) toward frontier-model competitiveness through a
sequence of numbered experiments. Each experiment is a self-contained
hypothesis: a change to the system, a measurable predicted outcome, and a
go/no-go criterion before moving to the next version.

**Before any experiment**, read:

1. `engine.py` — the core: programs, combinators, library, Bayesian update
2. `rce.py` — the CLI: train/chat/status/reset
3. `BENCHMARKS.md` — the running record of metrics across versions (you create
   and maintain this; if absent, create it on V1)
4. `LOG.md` — the running journal of every experiment, what changed, what
   happened, what to keep, what to revert (you create and maintain this)

**Architectural philosophy you must respect:**

- The system is a **library of explicit programs** weighted by Bayesian
  posterior with a Solomonoff length prior. Do not replace this with a
  neural net. That is the entire point of the project.
- Growth is by **recurrence (combinators)**, not search. New programs come
  from applying a small fixed set of combinators to existing ones. Do not
  introduce gradient descent over program space.
- **Hallucination must remain structurally impossible.** Every output is a
  sample from an explicit posterior. If you find yourself adding a generator
  that can fabricate, stop and reconsider.
- **Audit trails are sacred.** Every prediction must trace to specific
  weighted programs. Do not introduce opaque components.

**What you may add freely:**

- New primitive program types (more sophisticated predictors)
- New combinators (carefully — each one expands expressible space)
- New tokenizations (bytes are not the only choice)
- New training-data sources (HuggingFace datasets, books, code, etc.)
- Compute scaling (Modal for distributed training)
- Better posterior management (regularization, decay, replay)
- Evaluation harnesses (perplexity, downstream tasks, calibration tests)

**What you must justify in writing before adding:**

- Anything that breaks the audit-trail property
- Anything that introduces a non-Bayesian update rule
- Anything that requires a neural network
- Anything that increases library size by more than 10× in one step

## 1. How this is different from an LLM (and why the experiments target what they target)

| Axis | Frontier LLM | RCE |
|------|-------------|-----|
| Representation | Continuous tensor (~10¹² params) | Discrete program library |
| Training | Gradient descent on cross-entropy | Bayesian update + combinator generation |
| Inference | Single forward pass | Weighted vote across programs |
| Knowledge | Implicit, distributed in weights | Explicit, traceable to programs |
| Uncertainty | Logit calibration (unreliable) | Posterior entropy (computed exactly) |
| Hallucination | Emergent | Structurally impossible |
| Self-modification | None (fixed architecture) | Interpreter is a program; combinators can recur on themselves |
| Compute profile | Massive parallel matmul | Many small parallel program evaluations |
| Failure mode | Confident wrong answers | Refuses to answer (high entropy) |

The experimental program below is designed to attack each axis where the
RCE is currently weaker than an LLM (representational capacity, training
throughput, evaluation rigor) without sacrificing the axes where it is
stronger (auditability, calibration, self-modification).

## 2. Training philosophy (read carefully — this is NOT LLM training)

You are not doing gradient descent. You are not minimizing a scalar loss
across a held-out set with backprop. The training loop is:

1. **Stream bytes** (or tokens) from a corpus
2. **Each program in the library predicts** the next byte from recent context
3. **Bayesian posterior update**: programs whose distributions assigned
   higher probability to the actual byte gain log-weight; others lose
4. **Periodically: grow** — apply combinators to top-weighted programs to
   produce children
5. **Periodically: prune** — drop low-weight non-primitive programs
6. **Periodically: abstract** — when a sub-program appears in many
   high-weight programs, lift it to a new primitive (DreamCoder-style
   wake-sleep, but pure-symbolic)

There is no validation loss, no learning rate schedule, no warmup, no
checkpoint averaging. There is the posterior. The posterior IS the
training state. The library snapshot IS the model.

**Datasets:** HuggingFace datasets are perfect inputs. They are byte
streams (after tokenization or raw). Use `datasets` library to stream them.
Do NOT pre-shuffle into batches the way an LLM trainer does — that destroys
sequential structure the RCE needs. Stream in document-order.

Recommended HF datasets, by phase:

- **V1–V5 (toy):** `wikitext-2-raw-v1`, `tiny_shakespeare`, `agnews`
- **V6–V10 (small):** `wikitext-103-raw-v1`, `bookcorpus` subset, `the_pile`
  subset (~1GB)
- **V11–V15 (medium):** `slimpajama-6b`, `c4` English subset, `code_search_net`
- **V16–V20 (large):** `the_pile` full, `redpajama`, `dolma`

**Compute:** For V1–V8, a single laptop is fine. For V9+, use Modal:
- Modal lets you run Python functions on GPUs/CPUs in the cloud with
  zero infrastructure setup.
- The RCE does not need GPUs (no matmul). It needs many CPUs running
  many program evaluations in parallel.
- Modal's `@app.function(cpu=8, memory=16384)` decorator gives you that.
- For distributed training: use Modal's `.map()` to shard the corpus
  across workers, each maintaining a sub-library, then merge libraries
  on a master node by union + posterior renormalization.
- Custom domains (modal.com/docs/guide/webhook-urls#custom-domains) are
  useful in V18+ when exposing the chat endpoint as a service.

## 3. Plans V1 — V20 (in order; do not skip)

Each plan has: **goal**, **change**, **measure**, **gate** (must pass
before next version).

### V1 — Establish the benchmark harness
**Goal:** Be able to measure progress.
**Change:** Add `bench.py` with three metrics:
1. **Bits-per-byte (BPB)** on a held-out 100KB slice of `wikitext-2-raw-v1`.
   Compute: `-mean(log2(P(actual_byte | ctx)))` across the slice.
2. **Calibration AUC**: bin predictions by confidence, measure accuracy
   per bin, compute ECE (Expected Calibration Error).
3. **Refusal rate** on out-of-distribution inputs (random binary, foreign
   scripts not in training): fraction where entropy > 7.0.
**Measure:** Record baseline numbers for current system on `wikitext-2`.
**Gate:** Numbers reproducible across two runs within ±2%.

### V2 — Tokenization sweep
**Goal:** Bytes are not the only input. Try alternatives.
**Change:** Add a `Tokenizer` abstraction. Implement: bytes (current),
BPE (use `tokenizers` library, train a 4K vocab), word-level (split on
whitespace + punctuation).
**Measure:** BPB across all three tokenizations on `wikitext-2`.
**Gate:** Pick the tokenization with best BPB. Document why. (Hypothesis:
BPE will win because it shortens effective context and fits more pattern
in n-gram tables.)

### V3 — Richer primitive predictors
**Goal:** Current primitives are uniform, repeat, n-grams. Add stronger ones.
**Change:** Implement these primitives:
- `SuffixArrayPredictor`: full suffix array of training data; predicts by
  longest matching suffix (Markov of unbounded order). This is the
  Prediction-by-Partial-Match (PPM) family — known to be strong on text.
- `LZWPredictor`: build LZW dictionary; predict by dictionary entry probabilities.
- `KneserNeyNGram(n)`: n-gram with Kneser-Ney smoothing instead of add-k.
- `SkipGramPredictor(k)`: predict from non-adjacent context (e.g., bytes
  at offset -1, -3, -5).
**Measure:** BPB delta vs V2.
**Gate:** At least one new primitive beats best n-gram by >5% BPB.

### V4 — Combinator expansion
**Goal:** Four combinators is the minimum. Try more.
**Change:** Add three new combinators:
- `Gate(a, b, predicate)`: use `a` if a learned predicate fires on
  context, else `b`. Predicate is itself a small program.
- `Memo(a)`: cache `a`'s predictions on recently-seen contexts.
- `Mixture(a, b, c, ...)`: posterior-weighted mixture of >2 parents
  (current `Composed` is binary).
**Measure:** Library composition after training. Count how many
high-weight programs use each combinator type.
**Gate:** At least one new combinator appears in top-20 programs after
training. If not, the combinator is dead weight — remove it and document why.

### V5 — Wake-sleep abstraction
**Goal:** When a sub-program appears in many high-weight programs, lift
it to a new primitive. This is the engine of language growth.
**Change:** Add `abstract_phase()`: scan top-50 programs for shared
sub-trees in their combinator lineage; if a sub-tree appears ≥3 times,
create a new primitive that directly computes it (caching predictions),
add to library, and replace occurrences in parent programs with
references to the new primitive.
**Measure:** Library size over training time. With abstraction, library
size should asymptote even as data grows (compression).
**Gate:** Library asymptotes within 2× the size of V4 on the same data.

### V6 — Streaming HuggingFace ingestion
**Goal:** Move from local files to real datasets.
**Change:** Add `datasets/hf.py`. Implement streaming ingest from any
HF dataset by name. Use `datasets.load_dataset(name, streaming=True)`
and iterate. Add CLI: `python rce.py train --hf wikitext-103-raw-v1
--max-bytes 10_000_000`.
**Measure:** Throughput in bytes/second on a single laptop core.
**Gate:** Sustained ≥50KB/s ingestion without library blowup.

### V7 — Posterior decay and replay
**Goal:** Stationary Bayesian posteriors over-commit on early data.
The brain solves this with sleep-replay and synaptic homeostasis.
**Change:** Two additions:
1. **Decay**: every N bytes, multiply all log-weights by 0.99. Keeps
   posterior responsive to non-stationary data.
2. **Replay buffer**: sample 1K random past bytes per growth phase,
   re-run posterior updates on them. Prevents catastrophic forgetting.
**Measure:** Train on corpus A, then corpus B; measure BPB on A held-out.
With/without replay.
**Gate:** Replay version retains ≥80% of A's BPB after training on B.

### V8 — Parallel training on Modal
**Goal:** Single-laptop training won't scale to interesting datasets.
**Change:** Set up Modal app. Sharding strategy:
- Split corpus into N shards (one per worker).
- Each worker maintains its own library, trains independently.
- Master process periodically merges libraries: union of programs,
  log-weights summed where same program appears, then renormalize.
- Use Modal's volume mounts to persist libraries between runs.
**Measure:** Wall-clock time to process 100MB corpus on 1, 4, 16 workers.
**Gate:** Linear scaling within 30% (sub-linear is fine — true linear
would be miraculous given merge costs).

### V9 — First serious training run
**Goal:** Train at scale and see what happens.
**Change:** On Modal with 32 workers, train on 1GB of `the_pile`
(English subset). Use V2's best tokenization, V3+V4 primitives, V5
abstraction, V7 decay+replay.
**Measure:**
- Final library size
- BPB on held-out wikitext-103
- Top-20 programs by weight (log them all)
- Time-to-train
**Gate:** BPB ≤ 2.5 bits per byte on wikitext-103 held-out. (For
context: a small LSTM gets ~1.5; raw bytes are 8; the goal here is to
beat naive n-gram baselines decisively.)

### V10 — Calibration as a feature
**Goal:** The RCE's biggest structural advantage is honest uncertainty.
Make this measurable and exploit it.
**Change:**
1. Build a calibration dashboard: for each prediction during a held-out
   evaluation, plot stated confidence vs actual accuracy.
2. Add a `--strict` chat mode: refuse to generate any byte where the
   library's top prediction has confidence < 0.3.
3. Add a `--explain` chat mode: after each generated byte, print the
   top 3 programs that voted for it and their weights.
**Measure:** Strict-mode coverage (% of inputs it answers) vs accuracy.
LLMs cannot do this honestly — this is RCE's home turf.
**Gate:** ECE ≤ 0.05 (well-calibrated). Strict mode answers ≥30% of
prompts with ≥90% byte-level accuracy on the answered subset.

### V11 — Token-level evaluation against LLM baselines
**Goal:** Be honest about where we stand.
**Change:** Run the RCE on standard LLM benchmarks, treating it as a
language model:
- **LAMBADA**: predict last word of a passage
- **HellaSwag**: multiple-choice continuation (score each option's
  posterior probability under the library)
- **WikiText-103 perplexity**
- **TriviaQA** (closed-book): hopeless, but worth recording the
  refusal rate
**Measure:** Score on each. Compare to: random, n-gram baseline, GPT-2
small, GPT-2 large.
**Gate:** Beat n-gram baseline on ≥3 of 4 benchmarks. Do NOT expect to
beat GPT-2 yet — record the gap and the gap's slope across versions.

### V12 — Cross-domain training
**Goal:** A library trained on text-only is brittle. Diversify.
**Change:** Train on a mixture: 50% text (`c4`), 25% code
(`code_search_net`), 25% structured (`wikidata` triples). Use a
domain-tag prefix byte so combinators can branch on domain.
**Measure:** BPB per domain on held-out. Library composition by
domain (which programs activate where).
**Gate:** Per-domain BPB no worse than 1.2× the single-domain BPB.
(Multi-domain training shouldn't catastrophically harm any one domain.)

### V13 — Self-modifying interpreter
**Goal:** The architecture promises self-modification. Deliver it.
**Change:** Move the interpreter (the predict() dispatch logic) into
the program library itself, written in the same combinator language.
Allow combinators to apply to interpreter-programs, producing
candidate replacement interpreters. Run candidates in sandbox; adopt
if they reduce description length of the library OR speed up
inference without harming BPB.
**Measure:** Number of interpreter rewrites accepted across a long
training run. BPB and inference speed before/after.
**Gate:** At least one interpreter rewrite is accepted. (This is the
Gödel Machine moment — even one accepted self-rewrite validates the
architecture's central promise.)

### V14 — Active curiosity (intrinsic motivation)
**Goal:** Currently the system passively ingests whatever you feed it.
Add Schmidhuber-style curiosity: prefer training data that maximally
reduces prediction error.
**Change:** When connected to a streaming source with selectable
subsets, the trainer scores incoming chunks by **prediction error
under current library**. High-error chunks get prioritized in the
training queue. (This is the compression-progress drive.)
**Measure:** With curiosity on/off, learning rate (BPB drop per byte
of training).
**Gate:** Curiosity-on trains to target BPB in ≤70% the bytes of
curiosity-off.

### V15 — Conversational fine-grain
**Goal:** RCE chat output is currently noisy. Make it conversational.
**Change:**
1. Train on conversational corpora: `daily_dialog`, `persona_chat`,
   `oasst1`.
2. Add a `<dialog>` framing primitive that learns turn structure.
3. Generation: stop on turn-boundary tokens (newline + role marker).
4. Add `--continue` mode that maintains chat context across turns
   in the input.
**Measure:** Human evaluation on 50 prompts: rate coherence 1–5.
**Gate:** Mean coherence ≥3.0 (recognizable as on-topic, even if not
fluent).

### V16 — Knowledge persistence and provenance
**Goal:** Use the audit-trail advantage. Every fact the system "knows"
should be traceable.
**Change:** When a program achieves high posterior weight, log:
- The training shard(s) that drove its weight up
- The contexts where it consistently predicts well
- A "domain signature" (which byte n-grams it activates on)
Surface this in `status` and via a new `provenance <query>` command
that says: "When asked about X, programs P₁, P₂, P₃ activate, and
they were primarily strengthened by data from sources S₁, S₂."
**Measure:** Coverage: what fraction of high-weight programs have
non-trivial provenance.
**Gate:** ≥80% of top-100 programs have meaningful provenance traces.

### V17 — Distributed library federation
**Goal:** Why train one library when you can federate many?
**Change:** Build a library exchange format (gzipped pickle + signed
manifest of provenance). Two RCE instances can exchange libraries by
union, with weights renormalized and provenance preserved. Train
specialist libraries (one on code, one on math, one on dialogue),
then federate.
**Measure:** Federated library BPB on each domain vs specialist BPB
on its own domain.
**Gate:** Federation loses ≤15% BPB per domain vs specialist. (The
bet: combinator generation will discover cross-domain abstractions
that specialists couldn't.)

### V18 — Public chat endpoint via Modal
**Goal:** Expose to real users; gather real prediction-error signal.
**Change:** Modal web endpoint with custom domain. The chat endpoint
streams responses. Every conversation is logged with the user's
ratings (thumbs up/down per turn). Ratings feed back into curiosity
(V14) — high-disagreement turns become training-priority data.
**Measure:** Daily active conversations. Rating distribution.
Library growth from real-user data.
**Gate:** Sustained ≥10 conversations/day for one week. Library
grows ≥10% from user-generated curiosity signal.

### V19 — Multi-modal extension
**Goal:** Bytes are universal. Try non-text.
**Change:** Train on raw image bytes (small images, e.g., MNIST as
flat byte streams) and audio (8-bit μ-law samples). Same combinators,
same library.
**Measure:** BPB on images and audio. Visualize what programs the
library evolves for non-text data — do recur(recur(ngram)) chains
discover edge detection? Compositional structure?
**Gate:** BPB on images ≤ 6.5 (vs 8 for raw, vs ~4 for PNG).
This is a stretch goal; failure here is fine and informative.

### V20 — The honest report
**Goal:** Write up where we are. Be ruthlessly honest.
**Change:** Compose `RESULTS.md`:
- Final BPB on each benchmark
- Calibration metrics (where we likely beat LLMs)
- Cost per byte trained vs LLM equivalents
- Honest assessment of frontier-competitiveness
- What worked, what didn't, what's next
**Measure:** N/A — this is the report.
**Gate:** Document is publishable as a research note.

## 4. The Ralph loop (continuous trainer/director)

The Ralph loop is a long-running script that:
1. Reads the current version state from `state.json`
2. Loads this plan
3. Determines the next experiment to run
4. Sends instructions to a coding-capable LLM (Claude, via API)
5. Receives code/commits back, applies them
6. Runs the experiment
7. Logs results
8. Decides: gate passed → next version; gate failed → diagnose, retry,
   or escalate to human
9. Sleep, loop

The Ralph loop's prompt to the LLM each iteration is the next section.

## 5. The Ralph-loop prompt (this is what gets sent to the LLM each iteration)

```
You are Claude, acting as the engineer in a Ralph loop training the RCE.

CURRENT STATE:
- Version: {version}
- Last gate result: {pass|fail|in_progress}
- Last commit: {sha}
- Time since last human review: {hours}h

YOUR TASK THIS ITERATION:
Read these files in order:
  1. /workspace/rce/PLAN.md (this plan)
  2. /workspace/rce/LOG.md (running journal)
  3. /workspace/rce/BENCHMARKS.md (running metrics)
  4. /workspace/rce/engine.py
  5. /workspace/rce/rce.py
  6. Any V{version}-related files already started

Then determine:
  (a) What stage of V{version} we are at:
      - DESIGN: no code written for this version yet
      - IMPLEMENT: code partially written, not running
      - RUN: code complete, experiment in progress
      - EVALUATE: experiment complete, gate not yet checked
      - GATE_FAIL: gate failed, need to diagnose or retry
      - DONE: gate passed; advance to V{version+1}

  (b) The single next concrete action.

CONSTRAINTS:
  - Make ONE focused change per iteration. No mass rewrites.
  - Every change must be auditable: write to LOG.md what you did and why.
  - Every metric must go to BENCHMARKS.md as a row with date+version+score.
  - You MAY NOT introduce: neural networks, gradient descent,
    non-Bayesian update rules, or anything that breaks audit trails.
    If you believe an exception is warranted, write it in LOG.md as a
    PROPOSAL and stop. Do not implement until a human approves.
  - You MAY introduce: new primitives, combinators, tokenizations,
    datasets, evaluation harnesses, infrastructure (Modal), provenance
    tooling, calibration tooling.
  - If a gate fails twice with different fixes, escalate: write
    ESCALATE in LOG.md with diagnostic notes and stop.

OUTPUT FORMAT:
  1. STATE_ASSESSMENT: which stage you're in, what you read, what you found
  2. NEXT_ACTION: one specific change (code edit, command to run, file
     to create), with the exact diff or command
  3. EXPECTED_OUTCOME: what you expect to see after this action
  4. UPDATE_LOG: the line(s) to append to LOG.md
  5. UPDATE_BENCHMARKS: the row(s) (if any) to append to BENCHMARKS.md

Then perform the action via available tools. Then exit.

The Ralph loop will sleep, then call you again with updated state.
```

## 6. Bootstrapping the Ralph loop

To launch:

```bash
# Initial setup (human runs once)
mkdir -p /workspace/rce
cp engine.py rce.py PLAN.md /workspace/rce/
cd /workspace/rce
echo "version: 1" > state.json
echo "stage: DESIGN" >> state.json
touch LOG.md BENCHMARKS.md

# Launch the loop
python ralph.py --plan PLAN.md --interval 30m --max-iters 1000
```

`ralph.py` is a thin orchestrator: it calls Claude with the prompt above,
applies the returned actions, runs commands, captures outputs, updates
state, sleeps, repeats. It does not contain RCE logic. It is the
conductor, not the orchestra.

Reasonable safety rails for `ralph.py`:
  - Hard timeout per iteration (e.g., 60 min)
  - Disk quota check before each train run
  - Modal cost ceiling check before each cloud experiment
  - Required human review every 10 iterations OR on any ESCALATE flag
  - Append-only LOG.md (never rewrites history)

## 7. What success looks like at V20

You will have:
- A library of ~100K–1M small programs
- BPB on wikitext-103 between 2.0 and 2.5
- Calibration ECE ≤ 0.05 (likely better than any LLM, full stop)
- Honest refusal rate >40% on out-of-distribution inputs
- Demonstrable self-modification (interpreter rewrites accepted)
- Public chat endpoint with real users
- Compute cost orders of magnitude below an LLM of equivalent BPB
- A complete audit trail for every prediction

What you will NOT have:
- Frontier-LLM-level fluency on open-ended generation
- Strong performance on tasks requiring world knowledge the library
  hasn't explicitly compressed
- Anything resembling reasoning ability beyond what the combinator
  graph naturally produces

The plan is designed to test the central architectural bet: that
combinator-driven library growth is a real alternative scaling law,
not just a clever-looking dead end. By V20 we will know.

## 8. Final note to the executing LLM

This system is a serious bet on an unfashionable architecture. The
fashionable thing would be to give up and use a transformer. Don't.
The whole point is to find out whether the unfashionable path leads
somewhere interesting. Stay disciplined: one change per iteration,
measure everything, document everything, respect the architecture's
core promises (auditability, calibration, no hallucination), and let
the data tell us where we stand.

Begin with V1.
