**Mandatory pre-ingestion check** still applies — every new HuggingFace

reasoning corpora at V26, modality-specific data at V28); validate each
before training.

**Mandatory post-training coherence gate** (new in v2 — added because
v1.md V20 honestly established that low BPB doesn't entail coherent
output). After every V2x version's training run, run the coherence
harness against the freshly-saved library:

```bash
python coherence.py --lib <library.pkl> --version V<N> --tag <dataset-tag>
```

The harness exercises three test families against fixed prompt sets
that *do not change between versions* (the point is to see regressions):
1. **Refusal calibration** — in-dist prompts must NOT be refused;
   OOD prompts (random bytes, foreign script, byte-soup, emoji) MUST
   be refused. Score = mean of (in-dist accept rate, OOD reject rate).
2. **Generation quality** — sample N continuations from each in-dist
   prompt; score on UTF-8 validity, ASCII printable fraction, wordlist
   hit rate.
3. **Sample inspection** — captures fixed exhibits to `coherence_<V>_<ts>.json`
   and prints first samples to stdout for human eyeball.

**Coherence gate per V2x version:**
- `refusal_balanced_accuracy ≥ 0.85` (architecture's distinctive
  property holds)
- `ascii_printable_frac ≥ 0.80` (generated bytes are valid text, not
  garbled non-printables)
- **Must not regress vs the prior version's recorded coherence row in
  `BENCHMARKS.md`.** A version that improves BPB by silently breaking
  refusal or output validity is not progress — it's the V3-style
  silent regression v2.md exists to prevent. Treat regressions as
  GATE_FAIL until either fixed or explicitly waived in LOG.md with
  a written justification.

Programmatic single-prompt inference is also available for ad-hoc
testing:

```bash
python infer.py --lib <library.pkl> --prompt "..." --top-voters 3
```

`infer.py` auto-wraps a bare `Library` in V10's `Toplevel` so refusal
is always measured, never the V1 entropy heuristic.

## 1. What v1.md got right and what v1.md got wrong (lessons that shape v2)

This section exists because v2 is not a fresh plan — it's a corrective
and expansionary plan. Each lesson below cites specific V1-V20 evidence
and points at the V2x version that responds to it.

### 1.1. Mode collapse is the dominant failure mode of vanilla Bayesian update over a discrete library

V4 trained on 30KB. The posterior collapsed entirely onto `kn-5` (weight
1.0; every other program ≈0). V5's wake-sleep abstraction lifted **zero
programs** because top-50 were all direct children of `kn-5` — shallow
lineage, no shared non-primitive ancestor.

This is structural, not numerical. With `lr=0.03` over ~7500 steps, any
2× edge yields ~150 nats of log-odds. Once one program is consistently
best, the posterior is no longer an ensemble. v1.md's V7 (decay + replay)
is a band-aid; v2's V22 fixes the math.

**Response:** V22 — Posterior diversity as a hard constraint.

### 1.2. The refusal claim was a heuristic, not an invariant

V1's 100% refusal rate on `os.urandom` was a side-effect of n-gram
primitives returning uniform on missing context (entropy ≈ 8 bits). V3
introduced KneserNey, whose order-0 base case is the continuation
distribution — a learned unigram. Refusal collapsed to 0.0000.

The architecture's most-marketed property ("hallucination structurally
impossible") is currently false in code. Every downstream version
inherits the regression.

**Response:** V21 — Refusal as a structural invariant. **Must ship before
any V2x version that touches users.**

### 1.3. Combinators barely beat primitives at small scale

V1 trained at BPB 2.6209. The naive n-gram-5 baseline at V2 hit BPB
2.6744 on bytes. Two percent. The combinator framework — the central
architectural bet — contributed almost nothing at 100KB train scale.

This is suggestive, not damning. The v1.md plan banks on combinators
paying off at V9+ scale (1GB+ corpora). But it also points at a
structural issue: a *flat* library of heterogeneous predictors wastes
capacity by running every program on every context. Different predictors
should specialise on different contexts.

**Response:** V23 — Hierarchical library (MoE for symbols).

### 1.4. The BPE bet wasn't wrong, but was wrong at the scale we tested

V2's tokenisation sweep gave: bytes 2.67, BPE-4K 2.81, word 2.25. Word
won decisively. The plan's hypothesis ("BPE shortens context, fits more
pattern in n-gram tables") was reasonable but failed at 1MB — only 270K
BPE tokens populate a 5-gram table sparsely; smoothing penalty
dominates.

We deferred BPE to V11+ scale. v2 keeps the deferral live: V24's
compositional benchmarks include sub-word-aligned tasks where BPE may
reassert.

**Response:** No dedicated version, but V24 and V25 include a BPE
ablation as a sanity check.

### 1.5. Storage migrations gate scale

`plans/storage.md` describes three tiers. V6 triggered Tier-1 (LMDB).
V8 triggered the Modal sharding piece of Tier-1. V13's self-modifying
interpreter triggered the CoW addition. V16 triggered the provenance
store. Tier 2 (FoundationDB / TiKV / Redis hot tier) was scheduled
for V11-V15 but realistically gets pushed to v2.

**Response:** V23 ships the Tier-2 migration. V30 demands Tier-2 to be
production-ready.

### 1.6. The honest report (V20) is itself the deliverable

v1.md ends with V20: a research note. The temptation in a Ralph loop is
to skim past V20 and chase the next version. **Do not.** V20 is the
single most decision-relevant artefact in the plan. v2's branching in
§6 is gated entirely on what V20 reports.

**Response:** §6 of this file. Read V20's RESULTS.md before picking a v2
branch.

### 1.7. The validator gate is load-bearing

Pre-ingestion validation (introduced in v1.md §0) caught dataset issues
before compute was wasted. Keep using it. Every new HF dataset in v2
goes through `validate_dataset.py` first; failures get an alternate
dataset and re-validation. No exceptions.

## 3. Plans V21 — V30 (mainline; in order; do not skip)

Same format as v1.md §3: **goal**, **change**, **measure**, **gate**,
**dataset**, **storage migration** where applicable.

### V21 — Refusal as a structural invariant
**Goal:** Restore the "hallucination structurally impossible" claim
that V3 broke. Make refusal a property of the architecture, not a
side-effect of n-gram smoothing.

**Change:** Introduce a top-level Bayesian mixture:
```
Toplevel = Mixture(
    InDistribution = current_library,
    Background     = UniformPrimitive(vocab),
)
```
The Background's predict() returns uniform-over-vocab (entropy =
log2(|vocab|)). The Toplevel posterior is updated by the same Bayesian
rule as everything else. P(Background | ctx) is the **refusal
probability**, computed exactly per prediction. Generation refuses when
P(Background | ctx) > τ (default τ = 0.5).

This is architecturally cleaner than v1.md's entropy-threshold
heuristic because it's a *posterior* probability — the Background
weight is updated by likelihood ratio, not by hand-tuned threshold.
Audit trail: every refusal traces to "Background outvoted InDistribution
in this context, here are the actual likelihoods."

V3's KneserNey regression no longer matters: KN's confident-but-wrong
predictions on OOD inputs make InDistribution lose likelihood mass to
Background, which uniformly assigned 1/|vocab| to the actual byte. The
math now corrects what the smoothing primitives broke.

**Measure:**
1. Refusal rate on `os.urandom` (V1 baseline: 1.0; V3 regression: 0.0)
2. Refusal rate on Cyrillic text (Russian Wikipedia; foreign script not
   in train)
3. BPB on wikitext-2-raw-v1 held-out — must not regress vs V7 baseline
4. Calibration ECE on in-distribution — must not regress vs V10

**Gate:** refusal ≥0.95 on `os.urandom`; refusal ≥0.90 on Cyrillic;
BPB ≤2.5 on wikitext-2 held-out; ECE ≤0.05.

**Dataset:** wikitext-2-raw-v1 (in-distribution), os.urandom 20KB
(noise OOD), Russian Wikipedia 20KB via the [wikimedia/wikipedia](https://huggingface.co/datasets/wikimedia/wikipedia)
ru-config (script OOD).

**Storage migration:** none. The Background mixture is a one-line
addition to the top-level Library.predict; no new stores required.

**Justification for ordering V21 first:** every V2x version that
touches real users (V25, V27, V30) inherits whatever refusal property
holds. Shipping V25's LLM-as-primitive without V21 would produce a
system that confidently confabulates on OOD, which is the failure mode
the entire architecture exists to avoid. V21 is non-negotiable.

### V22 — Posterior diversity as a hard constraint
**Goal:** Eliminate mode collapse without depending on a hand-tuned
learning-rate schedule. Make the library a real ensemble again.

**Change:** Two interlocking changes to the posterior math:

1. **PAC-Bayes tempered posterior.** Replace the current
   `log_weight += lr * log_likelihood` with a tempered Gibbs posterior:
   `log_weight = (1/T) * sum_log_likelihoods + log_prior`. Temperature
   T ∈ [1, 16] is itself selected by maximising held-out BPB. T=1
   recovers current behaviour; T>1 flattens the posterior, preventing
   single-program domination. Mathematically defensible — Catoni 2007
   and the PAC-Bayes literature show tempered posteriors have better
   generalisation guarantees over discrete hypothesis classes.

2. **DPP-based top-k selection.** When `Library.top_k(n)` is called,
   instead of pure argsort, sample n programs from a Determinantal
   Point Process whose kernel is the program-prediction-signature
   similarity matrix. Penalises redundancy: two near-identical
   programs can't both be in top-k. This is a *selection* change
   (which programs we evaluate at inference), not a *posterior*
   change.

Combined effect: the posterior keeps real mass on multiple programs
(via tempering); the top-k actually evaluated stays diverse (via DPP).
V5's `abstract_phase` should now find shared sub-trees because top-50
will have richer lineage.

**Measure:**
1. Effective sample size of posterior: ESS = exp(entropy(posterior)).
   Current value at V4 mode collapse: ≈1. Target: ≥10.
2. V5's `abstract_phase` lift count over a 100K-step training run.
   V5 baseline: 0. Target: ≥3.
3. BPB on wikitext-2 — must not regress vs V21.
4. Library composition by combinator type — `gate` and `mix` (V4
   identified as removal candidates) should now reach top-20 if their
   case was real.

**Gate:** ESS ≥10; abstract_phase lifts ≥3 programs; BPB no worse than
V21; gate or mix appears in top-20 (validates v1.md's V4 hypothesis).

**Dataset:** wikitext-2-raw-v1 — same corpus as V4/V5 to isolate the
math change.

**Storage migration:** none — pure algorithmic change.

**Hyperparameter sweep:** T ∈ {1, 2, 4, 8, 16}. Pick T by held-out BPB
on wikitext-2 validation slice. Document the chosen T in `state.json`
and freeze for downstream V2x versions.

### V23 — Hierarchical library (MoE for symbols)
**Goal:** Stop running one flat ensemble of heterogeneous predictors.
Route predictions by context signature into specialist sub-libraries.
This is mixture-of-experts adapted to a discrete program substrate.

**Change:** Replace `Library` with `HierarchicalLibrary`:
```
HierarchicalLibrary = {
    router:          small Program that emits a sub-library id from ctx
    sub_libraries:   dict[id → Library]        # specialists
    cross_borrow:    set[Program]              # programs shared across subs
}
```

Routing decisions are based on context features that are *cheap* to
compute: domain-prefix bytes (if present), n-gram fingerprint over
last 32 bytes, character-class distribution (alpha vs digit vs punct
vs script). The router itself is a Program in the library — it's
weighted by how often its routing decision ended up matching the
sub-library that actually predicted best.

Bayesian update: route ctx → sub-lib → top-k from sub-lib → posterior
update on top-k → meta-posterior update on the router (did its choice
of sub-lib have the highest-likelihood prediction among all sub-libs?).

Combinator generation: per sub-library, plus periodic cross-library
borrowing where a high-weight program in sub-lib A is `Recurred` into
sub-lib B with reset weight.

This is also where v1.md V12's domain-tag prefix byte pays off — the
router uses V12's tag bytes as a strong signal.

**Measure:**
1. Per-domain BPB on V12's mixture (text 50% / code 25% / structured
   25%). Compare HierarchicalLibrary vs flat Library.
2. Routing entropy — how committed the router is. Should be low (≪1
   bit per decision on clear-domain inputs) but non-zero (some
   genuinely cross-domain queries).
3. Total inference latency — must not blow up vs flat library.
4. Cross-borrow utility: fraction of borrowed programs that earn
   meaningful weight in their new sub-library.

**Gate:** per-domain BPB improves ≥10% over flat library on V12
mixture; routing entropy ≤2 bits on clear-domain inputs; inference
latency within 1.5× of flat library.

**Dataset:** V12's mixture verbatim — c4 50%, code_search_net 25%,
intfloat/wikidata5m 25%. Reusing V12's mixture isolates the
hierarchical-library effect from a domain-shift confound.

**Storage migration (per `plans/storage.md` Tier-2):** **this is the
big one.** Migrate weights and output cache from LMDB to **FoundationDB**
(or TiKV — pick FDB if you can stomach the operational complexity, TiKV
if you want easier ops; document the choice in LOG.md). Add **Redis** as
hot tier for the most-requested ~10M (program_id, context) pairs.
Replace KenLM with **Marisa-trie** for n-gram primitives once the
billion-n-gram threshold is crossed. Per `storage.md` §"Tier 2", move to
**stratified Bayesian updates** (top 0.1% every byte, top 1% every 10
bytes, etc.) — required because hierarchical routing means each
sub-library gets fewer effective updates per byte and the long-tail
needs explicit stratification to not starve.

This is a major architectural change. **Per v1.md §0, write a PROPOSAL
in LOG.md, get human approval, then implement.** Do not migrate
speculatively.

### V24 — Compositional benchmarks (where RCE should win)
**Goal:** v1.md V11 measured RCE against LLM-shaped benchmarks
(LAMBADA, HellaSwag, WikiText perplexity, TriviaQA). Those reward
fluent surface generation, not symbolic composition. The architecture's
*bet* is that discrete program libraries beat continuous neural nets on
tasks where structured abstraction matters. V24 builds the benchmarks
that test that bet.

**Change:** Implement an evaluation harness for five benchmarks where
symbolic systems have historically beaten neural ones (or where the
evaluation axis explicitly rewards systematic generalisation):

1. **SCAN** (Lake & Baroni 2018) — commands → action sequences. The
   "add jump" and "length" splits are explicit compositional-
   generalisation tests. Symbolic systems hit 99%+; neural baselines
   collapse to <20% on the held-out splits.
2. **COGS** (Kim & Linzen 2020) — semantic parsing with held-out
   compositions. Same shape as SCAN, harder distribution.
3. **Out-of-distribution arithmetic** — synthetic. Train on 2-3 digit
   addition/multiplication; test on 5-7 digit. Length extrapolation
   is the explicit failure mode of transformers; symbolic systems
   that learn the *algorithm* generalise.
4. **Formal-grammar perplexity** — synthetic. Generate strings from
   a known PCFG; the ground-truth conditional distribution is
   computable. Calibration vs ground truth is the metric.
5. **Calibration under distribution shift** — train on wikitext-2,
   evaluate on out-of-domain wikitext-103 articles. ECE on shift is
   the metric.

**Measure:** scores on each. Compare to: random, n-gram baseline,
GPT-2-small, GPT-2-large, RCE-V20 (without any v2 changes), RCE-V23
(hierarchical, post-V21+V22+V23).

**Gate:** beat GPT-2-large on **≥2 of 5** benchmarks. SCAN length-split
is the most likely win; OOD arithmetic is the second-most-likely. A
single clean win on SCAN or arithmetic is the kind of result that
survives peer review and is independently informative regardless of
what V20's report said.

**Dataset:**
- SCAN: [brendenlake/SCAN](https://huggingface.co/datasets/brendenlake/SCAN) (use the `length` and `addprim_jump` splits)
- COGS: [IBM/COGS](https://huggingface.co/datasets/IBM/COGS) or the
  authors' GitHub mirror
- OOD arithmetic: synthetic, generated locally; document seed
- PCFG perplexity: synthetic, generated locally; ground-truth
  distribution stored alongside
- Calibration shift: [Salesforce/wikitext](https://huggingface.co/datasets/Salesforce/wikitext)
  `wikitext-103-raw-v1` held-out articles

**Storage migration:** none beyond V23.

### V25 — LLM-as-primitive (hybrid architecture)
**Goal:** RCE V24 establishes where pure-symbolic wins. Where it loses
(open-ended fluent generation; long-tail world knowledge), an LLM can
fill in *as one Program in the library*, weighted by the same Bayesian
update as everything else. The architecture's claims survive: the LLM's
contribution is measured, not assumed; if it's not pulling weight, it
gets pruned; if it's dominant, V21's Background mixture still controls
hallucination on OOD.

**Change:** New primitive class:
```
class LLMPrimitive(Program):
    model_name: str               # e.g. "gpt2", "phi-2", "tinyllama-1B"
    backend: Callable             # local inference; no remote calls
    top_k: int = 64               # truncate distribution at top-k
    cache: LRUCache               # context → distribution
    def predict(ctx: bytes) -> Counter[byte]:
        if ctx in cache: return cache[ctx]
        logits = self.backend(ctx, top_k=self.top_k)
        dist = softmax(logits)
        cache[ctx] = dist
        return dist
```

Constraints (these justify the neural-net intrusion permitted by §0):

- LLM runs locally via `transformers` or `llama.cpp`. **No remote API
  calls.** Audit-trail requires deterministic, inspectable outputs.
- Model is small enough for CPU inference: GPT-2-small (124M),
  Phi-2 (2.7B with int4 quant), or TinyLlama-1B. Document the choice.
- Bayesian update applies normally. If the LLM is consistently best,
  its weight grows; if not, it shrinks.
- The LLM is **not** allowed to be the router (V23) or the interpreter
  (V13). It is one voice in the choir, not the conductor.

**Justification for the neural-net exception:** v1.md §0 forbids
"anything that requires a neural network." V25 doesn't *require* a
neural network — the V20 system runs without one. V25 *adds* one as a
single ensemble member. The black-box predict() is wrapped in a Program
class with measured posterior weight. Every claim the architecture
makes (audit trail, calibration, structurally-impossible-hallucination
via V21) holds: the LLM is just another voice that gets weighted
Bayesianly and overruled by Background on OOD. If a future engineer
tries to extend this exception (e.g. "let the router be a neural net
too"), they must write a new PROPOSAL in LOG.md and get explicit human
approval. V25 is the only authorised neural intrusion.

**Measure:**
1. Posterior weight assigned to LLMPrimitive after V23-scale training
2. BPB on wikitext-103 with vs without LLMPrimitive in library
3. ECE — does adding the LLM degrade calibration?
4. V24's compositional benchmarks — does LLMPrimitive *help* or *hurt*
   on SCAN/COGS/OOD-arithmetic? (Hypothesis: hurts. The LLM will pull
   weight away from the symbolic programs that win those benchmarks.
   Measure to confirm.)
5. Refusal rate on Cyrillic — V21's Background mixture must still
   win out.

**Gate:** LLMPrimitive earns ≥0.05 posterior weight (real ensemble
member, not pruned, not dominant); BPB on wikitext-103 improves ≥10%
over RCE-V23-without-LLM; ECE ≤0.05; refusal ≥0.90 on Cyrillic.

**Dataset:** wikitext-103-raw-v1 train + held-out for BPB; V24's
compositional benchmarks for the helps-or-hurts ablation; Cyrillic for
refusal regression check.

**Storage migration:** add LLM weight files to the storage layout
described in `plans/storage.md`. They live in the code-store (they're
part of a Program) but at multi-GB sizes. Update Tier-2 storage budget
to reflect this; plan for ~2GB per LLMPrimitive.

### V26 — Reasoning combinators
**Goal:** v1.md V20 has predictive combinators (Compose, Branch, Mix,
Memo, Gate). Reasoning requires combinators that the V1-V20 set can't
express. V26 introduces four new combinators borrowed from the Lisp /
Scheme tradition. They are **symbolic**, not neural — they extend the
combinator algebra, not replace it.

**Change:** Four new combinators:

1. **Quoted(p)** — predict() returns a one-hot distribution over a
   *program-id token* (requires program-id tokens in the vocab; add a
   reserved range above the normal token range). Quoted(p) treats `p`
   as data.

2. **Eval(quote_p, ctx_p)** — predict() reads quote_p's emission to
   identify a program by id, then runs that program on the context
   produced by ctx_p. Together with Quoted, this gives the library
   reflection: programs can construct, store, and execute other
   programs.

3. **Search(p, k, scorer)** — beam-search over k candidate continuations
   produced by p, scored by a posterior-probability `scorer`. Returns
   the highest-scored continuation. This is how the system does
   look-ahead reasoning at inference time without a transformer's
   attention.

4. **Verify(p1, ..., pn, threshold)** — runs all n programs in parallel
   on the same context. If ≥threshold of them agree on the top
   prediction, emit it. Otherwise, pass control to V21's Background
   mixture (this couples Verify to the refusal invariant: cross-program
   disagreement is itself an OOD signal).

The interpreter from v1.md V13 is what makes Eval safe: candidate
interpretations of Quoted programs run in the V13 sandbox. CoW
semantics from `plans/storage.md` cover Eval-induced library mutations.

**Measure:**
1. bAbI tasks 1-20 (Weston et al. 2015) — deductive reasoning over
   simple textual scenarios. Small-LLM territory; symbolic systems
   have historically dominated.
2. GSM8K-style word problems — math reasoning. Stretch goal.
3. Library composition — how many top-50 programs use reasoning
   combinators after V23-scale training?
4. Inference latency — Search(beam=k) is k× more expensive than direct
   prediction. Document the latency cost.

**Gate:** solve ≥10/20 bAbI tasks (small-LLM territory); ≥1 reasoning
combinator (Quoted, Eval, Search, or Verify) appears in top-20 by
weight after training; latency cost of Search ≤5× direct prediction
(beam=4 default).

**Dataset:** [facebook/babi-tasks](https://huggingface.co/datasets/facebook/babi-tasks)
(English, 1k-train and 10k-train variants); [openai/gsm8k](https://huggingface.co/datasets/openai/gsm8k)
for the stretch.

**Storage migration:** V13's CoW semantics handle Eval-induced library
mutations. No new migration; verify that V13's rollback path actually
works by running 100 Eval-rewrite trials and confirming zero library-
state corruptions.

### V27 — Closed-loop curiosity from real users
**Goal:** v1.md V14 implemented synthetic curiosity (chunks scored by
prediction error). v1.md V18 deployed for real users with thumbs-up/
thumbs-down ratings. V27 fuses them: each user disagreement enters the
V14 curiosity queue with **provenance** (V16) attached, so retraining
traces to specific user / conversation / turn.

**Change:**
1. Webhook from V18 endpoint: thumbs-down event publishes a
   `(ctx, expected_byte, user_id, turn_id, conversation_id)` record
   to a Redis pub/sub channel.
2. Trainer subscribes; pushes to V14's curiosity queue with priority
   `recency × disagreement_score`. Disagreement score: 1 - P(rated
   continuation | library) at the time of generation.
3. Training pass on the queued items updates the posterior; provenance
   store (V16) records the program weight delta with cohort metadata.
4. New `provenance <user_id>` query returns: which programs were
   strengthened by this user's feedback; what fraction of their
   conversations they touched.
5. Privacy: user ids are hashed; raw ctx is hashed for storage; only
   the (hash, weight delta) pair persists past 30 days.

**Measure:**
1. Cohort-level BPB on user logs over 1-week windows — does training
   on disagreements actually reduce future disagreements?
2. A/B test: model_t (current) vs model_{t-1} (last week's snapshot)
   on next-day user satisfaction. Run 2-week parallel deployment.
3. Provenance coverage on user-driven program updates (extends V16's
   80% gate to user-cohort-level).

**Gate:** thumbs-up rate increases ≥5% week-over-week post-deployment;
A/B shows model_t wins on user satisfaction with p<0.05; provenance
coverage ≥80% on user-driven updates.

**Dataset:** real user logs from V18 (live). No new HF dataset, but
`validate_dataset.py` should be extended to emit a "vibes score" on
incoming user-traffic batches — corrupt user data is the new
expensive-failure mode at this stage.

**Storage migration:** add Redis pub/sub channel for thumbs events
(small; reuses the V23 Redis hot tier). Provenance store grows
linearly in user count; budget for ~100KB per active daily user.

### V28 — Modality-specific primitives
**Goal:** v1.md V19 tested bytes-of-images and bytes-of-audio with the
same combinator set. The gate (BPB ≤6.5 on images) was a stretch; the
plan said failure here was "fine and informative." V28 commits to
modality-specific primitives if and only if V19 hit the stretch goal.
Otherwise V28 is **skipped** and the resources go to V29.

**Change (conditional on V19 success):**

1. **ImagePatchPrimitive(patch_size)** — analogous to NGramPrimitive
   but in 2D. Predicts pixel value from a spatial neighbourhood of
   given patch size. Trained on flat byte streams of small images
   (MNIST 28×28 = 784 bytes per image); the primitive internally
   reshapes to 2D for prediction.
2. **AudioFramePrimitive(window, sample_rate)** — predicts amplitude
   from a temporal context window plus FFT-derived spectral features.
   Same Bayesian-update interface as every other primitive.
3. **Modality routing** via V23's HierarchicalLibrary — image and
   audio context features dispatch to modality-specific sub-libraries.
   Cross-borrow allows compositional combinators (Compose, Recur, Mix)
   to migrate insights across modalities; modality-specific primitives
   stay siloed.

**Conditional skip rule:** if V19's BPB on MNIST was >7.0 (i.e.
near-trivial improvement over raw bytes), the architecture's
multi-modal claim is too weak to invest in. Skip V28; document in
LOG.md why; allocate the budget to V29.

**Measure (if not skipped):**
1. BPB on MNIST with ImagePatchPrimitive vs V19's bytes-only baseline.
2. BPB on LibriSpeech (μ-law 8-bit) with AudioFramePrimitive vs V19.
3. Cross-modal compositionality: does training on text+images produce
   any combinator that activates on both? (Hypothesis: yes for the
   Recurred and Composed combinators; no for primitives.)

**Gate:** BPB on MNIST ≤5.0 (V19's stretch was 6.5); BPB on
LibriSpeech ≤6.0; ≥1 combinator activates on both text and image
contexts.

**Dataset:** [ylecun/mnist](https://huggingface.co/datasets/ylecun/mnist),
[openslr/librispeech_asr](https://huggingface.co/datasets/openslr/librispeech_asr)
(`clean-100` config, μ-law downsample). Same as V19 to isolate the
primitive change.

**Storage migration:** ImagePatchPrimitive and AudioFramePrimitive
each carry a fitted parameter set (the spatial / temporal n-gram
tables). These are per-primitive caches; reuse the Tier-2 LMDB / FDB
weight store. No new tier required.

### V29 — Theory paper
**Goal:** By V28, RCE has substantial empirical results. Before
allocating the budget needed for V30 (production deployment), formal
grounding is required for the architectural claims. V29 is a research
deliverable — a paper, not a code commit.

**Change:** Compose `THEORY.md` (and target submission to NeurIPS /
ICML / ICLR's program-induction or symbolic-AI track). Theorems to
prove or refute:

1. **Universality** — given the V1-V20 primitive set and the
   combinator algebra (extended through V26), what class of
   conditional distributions is expressible? *Conjecture: any
   computable P(byte | ctx), with description-length bounded by a
   constant times the Kolmogorov complexity of the distribution.*
2. **Posterior-convergence rate** — given iid data drawn from a
   distribution in the expressible class, how fast does the Bayesian
   posterior over the discrete library concentrate on the optimal
   program? *Conjecture: O((log N) / N) regret in the Solomonoff-prior
   regime; PAC-Bayes-style bound.*
3. **BPB lower bound** — information-theoretic lower bound on
   achievable BPB given training-data size and library size. Compare
   to RCE V20 and V23 actuals. The gap (achieved minus lower bound)
   tells us how much architecture-specific waste is left.
4. **Calibration guarantee** — given V21's Toplevel mixture with
   Background, formal calibration bound (PAC-style or martingale).
   *Conjecture: ECE ≤ O(1/√N) for in-distribution, exactly bounded by
   Background prior on OOD.*
5. **Mode-collapse no-free-lunch** — formalise V22: for what posterior
   schemes (untempered / tempered / DPP-selected) is mode-collapse
   provable, and at what rate?

**Measure:** theorems proven or refuted; if submitted, peer-review
feedback. Negative results (refuted conjectures) are publishable and
informative — they tell the field where the architecture genuinely
fails.

**Gate:** at least 2 of 5 conjectures proven (or formally refuted with
a counterexample); paper submitted to a venue. The Ralph loop should
escalate to a human at V29 — theorem-proving is not a Ralph-shaped
task. Ralph's role at V29 is to gather empirical evidence (run the
ablations the proofs reference), maintain THEORY.md as the human
researcher writes, and verify claims against benchmark data.

**Dataset:** N/A (theoretical). Empirical references cite V1-V28
data; cite specific BENCHMARKS.md rows.

**Storage migration:** none.

### V30 — Production deployment, narrow-domain
**Goal:** V20's RESULTS.md likely shows RCE losing to LLMs on fluent
generation but winning on calibration and audit trail. V30 commits to
"calibration as product" — deploy in a narrow domain where confident
confabulation is dangerous and "I don't know" is more valuable than a
fluent guess.

**Change:** This is a deployment, not a code commit. Pick one domain;
partner with an organisation that needs calibrated AI; deploy via
Modal endpoint; measure real-world impact.

Domain candidates (in order of fit):

1. **Education** (math tutoring, programming feedback) — showing
   reasoning beats showing answers; provenance helps explain *why* an
   answer is suggested; refusal on out-of-curriculum questions is
   acceptable behaviour.
2. **Legal research** (citation chains, statute lookup) — provenance
   maps directly to citation requirements; "no answer" with a refusal
   reason is the regulatory-acceptable failure mode.
3. **Clinical decision support** (high regulatory bar, requires
   partnership with a medical institution; calibrated refusal is
   genuinely life-critical).
4. **Financial compliance** (audit-trail required by regulators;
   provenance is a feature not a debug tool).

The pilot domain is whichever finds a willing partner first.
Education is the lowest regulatory bar and the most-likely first pilot.

**Constraints:**

- Training data must be appropriately licensed. V12's wikidata5m,
  V15's oasst1, V19's MNIST/LibriSpeech, the V24 SCAN/COGS benchmarks
  — all permissively licensed. Domain-specific data (medical, legal)
  must clear licensing review with the partner organisation **before**
  any training compute is spent.
- Refusal rate (V21) is the headline metric. Fluency is secondary.
  The product positioning is "I tell you when I don't know," not "I
  generate convincing text."
- Provenance (V16) must satisfy the partner organisation's regulatory
  or audit requirements. Document the exact requirements before
  deployment; verify the V16 + V27 stores meet them.
- All deployment code re-uses V18's Modal endpoint pattern; no new
  serving infrastructure unless the partner explicitly requires it.

**Measure:**
1. Domain-specific accuracy (evaluation criterion negotiated with
   partner)
2. Refusal rate (must be ≥30% on hard prompts; this is a feature)
3. User trust (longitudinal study, partner-provided)
4. Cost per query (compare to GPT-4 baseline; the bet is RCE wins on
   total cost despite higher per-query CPU)

**Gate:** pilot organisation signs a renewal — i.e. agrees to keep
running the system after the trial period. This is the real-world
gate; academic metrics don't matter if no one will use the product.

**Dataset:** domain-specific, partner-provided. Pre-ingestion
validation (`validate_dataset.py`) still applies — failure here means
the partner's data has issues that must be resolved before any
deployment compute is spent.

**Storage migration:** Tier-2 storage stack must be production-ready
by V30. If V23's migration is still in PROPOSAL state, V30 is blocked
until it lands.

## 4. The honesty checklist (gate semantics for v2)

v1.md's gates are mostly numerical (BPB thresholds, library sizes,
calibration ECE). v2's gates are a mix of numerical and judgement-
based. Each V2x version's gate must be evaluated against this
checklist before advancing:

1. **Did the change actually produce the predicted effect, or did
   something else change?** Mode collapse on `kn-5` (V4) wasn't in the
   v1.md plan; it was a discovered failure. v2 versions must
   distinguish "gate passed because the hypothesis was right" from
   "gate passed for a reason we don't understand."
2. **Are the architectural invariants intact?** Refusal (V21),
   posterior diversity (V22), audit trail (V16+V27). After every V2x
   version, run the regression suite that checks these. v1.md V3's
   silent refusal regression must not happen again.
3. **Is the LOG.md entry honest about what failed?** v1.md's V4-V5
   entries documented mode collapse plainly. v2 must continue this.
   "Gate passed" with no caveats is suspicious; gate-passes that hide
   regressions are worse than gate-failures.
4. **Is the V2x change consistent with the V20 RESULTS.md regime?**
   If RESULTS.md said "Loss" and you're chasing V25's hybrid, stop
   and re-read §6.

## 5. The Ralph loop's behaviour in v2

v1.md §5 specifies the Ralph-loop prompt. v2 makes four changes:

1. **Required reading list** is updated. Ralph now reads, in order:
   v1.md, v2.md, RESULTS.md, LOG.md, BENCHMARKS.md, engine.py, rce.py.
2. **Gate semantics** include the §4 honesty checklist. A
   gate-passed-with-caveats triggers a LOG.md entry and a brief human
   review request, not an automatic advance.
3. **Theorem versions** (V29) trigger automatic human escalation.
   Ralph's role at V29 is bounded; the human-researcher role is
   primary.
4. **Coherence gate runs after every training step** (per §0). The
   loop's per-version cycle is:
       a. design / implement the version's change
       b. run training (per the version's `Change` and `Dataset`)
       c. run `coherence.py` against the freshly saved library
       d. run the version-specific `Measure` step (BPB, ECE, etc.)
       e. evaluate the version's `Gate` AND the coherence gate
       f. only advance if BOTH gates pass; otherwise diagnose, retry,
          or write a waiver in LOG.md
   Sample inference exhibits captured by step (c) feed into the
   §4 honesty checklist's "what failed" question.

Otherwise, v1.md §5's prompt template, §6's bootstrap protocol, and
§7's safety rails (timeout, disk quota, Modal cost ceiling, append-only
LOG.md) all carry forward unchanged.

## 6. Branching on V20's RESULTS.md

When V20 lands, read RESULTS.md before deciding which v2 path to
follow. The mainline above assumes the **Mixed** regime. The
alternatives:

### 6.1. Win regime
**Indicators in RESULTS.md:** BPB ≤2.0 on wikitext-103; ECE ≤0.05;
≥1 interpreter rewrite (V13) accepted; SCAN-style baseline already
beating GPT-2 in V11.

**v2 path:** V21 → V22 → V23 → V24 → **skip V25** → V26 → V27 →
(V28 conditional) → V29 → V30.

**Rationale:** the architecture works. An LLM crutch (V25) would
dilute the win. Push hard on V23 (scale) and V30 (deployment); make
V29 a triumphant theory paper.

### 6.2. Mixed regime (mainline)
**Indicators:** ECE ≤0.05; BPB plateaus at 2.5–3.0 on wikitext-103;
interpreter rewrites flaky; loses to GPT-2 on most V11 benchmarks.

**v2 path:** full mainline V21 → V30 as written above.

**Rationale:** calibration is genuinely a win; coverage is genuinely
a loss; V25's hybrid is the honest fix; V30 deploys where calibration
is the product.

### 6.3. Loss regime
**Indicators:** BPB > 3.0; mode collapse persists despite V7+V22;
abstraction never lifts; SCAN/COGS results no better than n-gram
baseline.

**v2 path:** V21 → V22 → V29 (early; theory paper becomes the
deliverable) → V30 narrowed to a single high-stakes domain.

**Rationale:** the empirical bet didn't pay off. The architecture's
calibration and audit-trail properties may still justify deployment
in narrow regulatory domains, but the scaling story is dead. V29
becomes the wind-down: prove or refute the conjectures honestly,
publish, redirect future work.

**Caveat:** even in Loss regime, V21 and V22 must ship. The
architectural invariants (refusal, posterior diversity) are the
foundation of any honest story about what RCE is. Shipping a "we
tried, it didn't work" paper without first fixing the V3 regression
would be dishonest about what was actually tested.

### 6.4. None of the above
If RESULTS.md doesn't fit any of the three regimes — escalate. Write
the ambiguity in LOG.md, stop, ask a human. The Ralph loop is not
qualified to interpret a genuinely novel V20 outcome.

## 7. What success looks like at V30

By V30, in the Mixed regime, you will have:

- A library whose refusal rate is a measured posterior probability,
  not an entropy heuristic
- A posterior that doesn't collapse onto a single program
- A hierarchical, routed library that scales to hundreds of millions
  of programs on a Tier-2 storage stack
- Compositional benchmarks (SCAN, COGS, OOD arithmetic) where RCE
  beats GPT-2-large on at least two
- A deployed, paying pilot in a narrow domain where calibrated
  refusal is the product
- A research paper formalising at least two of the five architectural
  conjectures
- An audit trail that traces every prediction, every program weight,
  every user-driven update — back to specific training shards,
  specific user feedback, specific provenance entries

What you still won't have:

- Frontier-LLM-level fluency on open-ended generation
- A mass-market chat product
- Resolution of the trillion-program scaling question — that's
  Tier-4 territory in `plans/storage.md` and remains an open research
  problem

The plan is designed so that **even in the Loss regime**, the
deliverables are valuable: an honest theory paper, a working
calibrated-refusal product in one narrow domain, and a public
demonstration that the unfashionable architecture was tested
rigorously and failed in specific, characterisable ways. Either way,
the field learns something.

## 8. Final note to the executing LLM

v1.md's final note told you to stay disciplined and not give up on the
unfashionable architecture. v2.md tells you something different:
**stay honest**.

By V20 you will have evidence — empirical, mechanistic — about whether
combinator-driven library growth is a real alternative scaling law. The
temptation in a Ralph loop is to keep shipping versions whether or not
the evidence supports them. Don't.

If V20 says the architecture works, push V21-V30 confidently. If V20
says it doesn't, ship V21-V22 (the architectural invariants must be
honoured regardless), then write the honest paper, then deploy in the
narrow domain where calibration alone justifies the product. Either
outcome is a real result.

The original sin to avoid is V3's silent refusal regression: a gate
passed, a benchmark improved, and a foundational architectural claim
quietly broke. v2 exists in part to repair that. Don't repeat it.

Begin with V21.
